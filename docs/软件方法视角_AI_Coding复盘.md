# 《软件方法》视角下的 AI Coding 复盘：GIS Data Agent 实践

> 作者自我复盘笔记 | 2026-04-06
>
> 理论框架：潘加宇《软件方法》ABCD 工作流
>
> 项目：GIS Data Agent (ADK Edition) v18.5

---

## 0. 引言

2026 年 2 月 18 日，我用 Claude Code 写下了 GIS Data Agent 的第一行代码。47 天后的今天（4 月 6 日），它已经是一个拥有 13.8 万行 Python 代码、3300+ 测试、254 个 REST API、40 个工具集的 AI 地理空间分析平台。264 个 commit，从 v1.0 到 v18.5。

速度令人兴奋，但兴奋过后该问一个严肃的问题了。

潘加宇老师在《软件方法》中提出了软件开发的四个建模工作流：

- **A-业务建模**：定位需要改进的目标组织以及该组织最需要改进的问题
- **B-需求**：描述为了改进组织的问题，待引入的信息系统必须具有的整体表现
- **C-分析**：提炼为了满足功能需求，待引入的信息系统需要封装的核心域机制
- **D-设计**：考虑质量需求和设计约束，将核心域机制映射到选定非核心域上实现

以及那个核心公式：**利润 ＝ 需求 － 设计**

AI Coding 工具（Claude Code、Cursor 等）本质上是在 D 工作流上发力。但正如潘老师警告的："由于没有把控住 C-分析的环节，随着系统需求的增加，AI 的各种修补、拼凑变得越来越笨拙，最终不可收拾。"

这篇复盘就是要用 ABCD 框架，诚实地审视这 47 天里我做对了什么、做错了什么。

## 1. ABCD 各工作流的实际执行情况

### 1.1 A-业务建模：做了，但顺序有问题

#### 做得好的

我确实做了 A 工作流的工作，而且用了《软件方法》的方法论。

**（1）定位目标组织规格**

`docs/软件方法_定位目标组织规格.md` 中，我按照潘老师课件中的提示词模板，输入了开发组织的能力集合（GIS 应用开发-优秀、自然资源领域经验-优秀、行业客情关系-卓越），让 AI 推导目标组织规格。这一步是严格遵循"从开发组织能力出发定位目标组织"的方法论的。

**（2）企业架构分析**

`上海测绘院_4A企业架构设计总览.md` 做了四层企业架构分析（业务架构、应用架构、数据架构、技术架构），明确了目标组织从"地图生产者"向"时空数据服务商"转型的业务愿景。

**（3）业务流程改进**

`docs/软件方法_改进业务序列图.md` 按照潘老师讲的三种改进模式——物流变信息流、改善信息流转、封装领域逻辑——分析了测绘质检的现有流程和改进方向。

`上海测绘院_BA业务架构细化方案.md` 设计了三条业务价值流（感知获取流→治理加工流→赋能服务流），定义了六个核心业务职能域。

**（4）竞品分析和市场定位**

PRD 中的 Problem Statement 清晰地识别了市场断层："一边是强大但门槛极高的专业 GIS 工具，另一边是普及但不具备空间分析能力的通用 AI 工具"——这是一个典型的 A 工作流产出：识别目标组织的痛点和改进机会。

#### 做得不好的

**（1）时间顺序倒置：先有 D 再补 A**

查看 git 记录，项目的第一个 commit 是 2026-02-18（直接开始编码），而 `软件方法_定位目标组织规格.md` 和 `上海测绘院_4A企业架构设计总览.md` 是在系统已经有了 v14+ 之后才做的（约 2026-03-25 之后）。

这就是潘老师批评的**思维颠倒**——先写代码，再"补文档"。虽然我后来补做的 A 工作流质量不差，但它已经不是在引导 D，而是在追认 D。真正影响系统走向的决策，是在 2 月份那个"拍脑袋"的下午做出的。

**（2）视角混淆：为谁做 A？**

`软件方法_定位目标组织规格.md` 中用的是"北京超图软件股份有限公司自建数产品线"的身份。这说明这个 A 工作流是为**客户的产品线**做的战略定位，而不是为我自己的项目做的。换句话说，我在帮客户做 A，但自己的项目反而缺少一个"我为什么要做这个项目、我的目标组织是谁"的显式推导。

PRD 中虽然有市场分析，但那更接近一份"投资人看的材料"，不是严格的 A 工作流产出。

**（3）缺少"as-is"到"to-be"的量化基线**

业务建模做了定性分析（愿景、价值流、改进模式），但没有做量化基线。比如：测绘质检的当前错检率是多少？人工质检一份数据需要多长时间？引入系统后的目标指标是什么？没有基线，改进就无法度量。

### 1.2 B-需求：做得最好的一环

#### 做得好的

**（1）PRD 先行**

`PRD_GIS_Data_Agent_V1.md` 的日期是 2026-02-22，而密集编码是从 2026-02-26 开始的。这说明**需求确实是在主要编码之前写的**，不是事后补的。这是整个项目中做得最符合 ABCD 顺序的一个环节。

PRD 的质量也不错：

- **Problem Statement**：清晰，一句话说明断层在哪
- **5 个产品目标（G1-G5）**：每个目标都有可量化的衡量标准
- **6 条 Non-Goals**：明确"不做"什么——这很关键，因为大部分项目只说做什么，不说不做什么
- **4 个用户画像 + 12 个用户故事**：从数据分析师、GIS 从业者、企业决策者、学生研究者四个角度写，每个故事都有 P0/P1/P2 优先级
- **成功度量指标**：区分了领先指标（time-to-first-wow、意图识别准确率）和滞后指标（注册量、企业签约数、NPS）

**（2）QC Agent 的需求 Gap 分析**

`docs/surveying_qc_agent_gap_analysis.md` 做了一份有深度的需求分析——拿着客户的《测绘质检智能体建设工作方案》原文，逐条拆解七大能力需求，对照现有系统能力做 gap 矩阵。这是正经的 B 工作流：从涉众的文档中提炼系统需要的功能。

#### 做得不好的

**（1）PRD 之后，需求退化成了功能列表**

PRD 只覆盖了 v1.0 的范围。从 v12 到 v18.5 的所有后续需求，基本都写在 `docs/roadmap.md` 里，格式是 markdown checklist：

```
- [x] feat(v17.0): Fusion v2.0 — temporal alignment, semantic enhancement
- [x] feat(v17.1): vector tile rendering — 3-tier adaptive delivery
- [x] feat(v18.0): DB optimization — asyncpg + connection pool 5→20
```

这不是需求，这是**功能列表**。潘老师严厉批评过这种做法——"想到什么写什么，边界跳来跳去"。

**（2）Roadmap 中需求和设计混杂**

Roadmap 中 "asyncpg 异步引擎"、"连接池 5→20"、"物化视图" 这些是 D（设计/实现），不是 B（需求）。把需求和设计写在同一个列表里，后果就是分不清哪些是"提升销售"（B），哪些是"降低成本"（D）。

更严重的是，roadmap 不断对标新的竞品——SeerAI、DeerFlow、AgentArts、Datus.ai——每次对标就新增一批 checklist 条目。这导致 B 不断扩张但缺乏收敛，系统从一个"空间数据分析智能体"逐渐膨胀成一个试图覆盖所有能力的"平台"。

**（3）缺少用例规约**

PRD 写了用户故事（"作为数据分析师，我想上传一份 Excel 文件……"），但从来没有写过正式的**用例规约**——用例名、前置条件、主成功场景步骤、扩展场景、业务规则。用户故事是 B 的起点，不是 B 的全部。没有用例步骤，D 层就只能靠 AI 自己"猜"应该怎么实现交互流程。
### 1.3 C-分析：最薄弱的环节

这是整篇复盘最核心的部分。C-分析是"提炼核心域机制"——系统要封装的领域概念、概念之间的关系、领域规则。它独立于实现技术，是连接"卖什么"（需求）和"怎么做"（设计）的桥梁。

#### 做得好的

**（1）GIS 本体预定义**

`data_agent/standards/gis_ontology.yaml`（154 行）定义了 37 个语义等价组（area↔面积↔mj↔zmj↔TBMJ 等）、12 条派生规则、6 条推理规则。这是**在编码之前**做的领域概念形式化，属于正经的 C 工作流产出。

**（2）质检缺陷分类法**

`data_agent/standards/defect_taxonomy.yaml`（320 行）按照 GB/T 24356-2009 标准定义了 30 个缺陷编码、5 大类（格式/精度/拓扑/缺失/规范）、3 级严重程度，还标注了每种缺陷是否可自动修复。这是从领域标准出发的概念建模，不是从代码反推的。

**（3）四层元数据模型（概念）**

v18.0 设计的四层元数据架构——技术元数据、业务元数据、运维元数据、血缘元数据——概念上是清晰的领域分层。

#### 做得不好的（核心问题）

**（1）从头到尾没有画过一张领域类图**

这是最大的问题。47 天、264 个 commit、13.8 万行代码，但从来没有做过一张核心域概念模型图。

DataAsset（数据资产）、SpatialEntity（空间实体）、QCDefect（质检缺陷）、Workflow（工作流）、Skill（技能）、User（用户）、Pipeline（管线）——这些核心域概念之间是什么关系？哪些是聚合根？哪些有泛化层次？生命周期如何关联？

没有人回答过这些问题。每次新加功能，AI 就在 D 层直接新建一张 DB 表，但这张表和已有的表之间的领域关系从未被显式思考过。

潘老师说的"需求→设计一一映射"反模式，在这个项目里精确复现了：

```
需求："支持数据资产管理"  →  新建 agent_data_catalog 表
需求："支持元数据治理"    →  新建 agent_data_assets 表（又一张！）
需求："支持语义注册"      →  新建 agent_semantic_registry 表
```

三个需求产生了三张各自独立的表，但它们在领域概念上其实是**同一个聚合（数据资产）的不同侧面**。由于没有 C 层的领域模型来约束，D 层就各做各的。

**（2）52 个 migration = 52 次 schema 补丁**

数据库的 52 个 migration 文件是一部"有机生长史"：

| 时段 | Migration | 特征 |
|------|-----------|------|
| v1 (002月) | 001-007 | 基础表（users, memories, audit） |
| v2 (003月初) | 008-018 | 语义、目录、工作流（随功能新增） |
| v3 (003月中) | 019-030 | 知识图谱、技能、虚拟数据源 |
| v4 (003月底) | 031-043 | RLS、数据分级、质检复核 |
| v5 (004月) | 044-052 | 元数据统一、TD-001 修复 |

这 52 个 migration 不是从一个领域模型分阶段落地的产物，而是"每加一个功能就补一张表"的结果。如果一开始有一张领域类图，很多表的字段和关系在 v1 就该定义好，而不是到 v5 才发现需要统一。

**（3）技术债 TD-001：双数据资产表**

这是 C 层缺失的最典型后果。

- `agent_data_catalog`（migration 008）：最早的数据目录表，字段偏技术描述
- `agent_data_assets`（migration 044）：后来设计的数据资产表，有四层元数据 JSONB
- `migration 048`：不得不写一个统一迁移脚本，把旧表数据搬到新表

如果一开始做了 C-分析，画了"数据资产"这个核心域概念的类图（包含哪些属性组、和其他概念什么关系），就不会出现两张表并存的尴尬。这个技术债直到 v15.8（2026-03-29）才被修复，此时系统已经开发了 40 天。

**（4）Toolset 按实现便利组织，而非按领域概念**

项目有 40 个 toolset，但大部分是按**技术活动**分组的：

- ExplorationToolset（探索）
- GeoProcessingToolset（地理处理）
- VisualizationToolset（可视化）
- AnalysisToolset（分析）
- StreamingToolset（流式传输）

这就像把人体器官按"输入器官"、"处理器官"、"输出器官"分类——这是功能分类，不是领域分类。

只有后来新增的几个 toolset（GovernanceToolset、DataCleaningToolset、PrecisionToolset）才是按**领域概念**组织的。而这些恰好是做了 QC 需求 gap 分析之后才加的——也就是说，**做了 B 的项目，C 的质量就明显高于没做 B 的项目**。

**（5）多个"真相源"并存**

语义元数据至少有三个来源：

1. `semantic_layer.py` 的内存缓存（5 分钟 TTL）
2. `standards/gis_ontology.yaml` 的静态 YAML 定义
3. `agent_semantic_registry` 数据库表

谁是权威？发生冲突时以哪个为准？没有一个统一的领域模型来回答这个问题。代码里是用 fallback 链来处理（先查缓存→查 DB→查 YAML），但这是 D 层的权宜之计，不是 C 层的设计决策。

### 1.4 D-设计/实现：AI Coding 的主战场

#### 做得好的

**（1）产出速度**

47 天，264 个 commit，13.8 万行 Python 代码，3319 个测试（148 个测试文件），254 个 REST API，40 个 toolset，18 个 ADK Skill，完整的 CI/CD 流水线。

无论如何评价这个项目的其他方面，这个产出速度在没有 AI Coding 工具的时代是不可想象的。

**（2）测试质量**

测试不是凑覆盖率的。以 `test_fusion_engine.py` 为例：有完整的 fixture 工厂（`_make_vector_fixture`、`_make_tabular_fixture`、`_make_raster_fixture`），mock 目标精准（在导入点 mock 而非源头），测试场景覆盖正常路径和异常路径。

CLAUDE.md 里专门记录了每个模块的 mock 目标（"Mock targets in tests: Always mock `get_engine` at the import site"），这保证了测试的一致性。

**（3）模块分离的基本面**

- `db_engine.py`（100 行）：单例连接池，读写分离预埋，所有模块通过 `get_engine()` 获取——不允许直接 `create_engine()`
- `toolsets/` 目录：38 个文件，每个 toolset 独立注册
- `user_context.py`：用 `ContextVar` 做用户身份传播，不靠全局变量
- `intent_router.py`：从 `app.py` 抽取出来的意图分类，职责单一

**（4）CI/CD 完整度**

GitHub Actions 配置了 PostGIS 集成测试、前端构建、Agent 评估流水线。有 staging 和 production 两套部署工作流。可观测性从第一天就有（Prometheus 指标、结构化日志）。

#### 做得不好的

**（1）God Module 问题**

`app.py` 有 3872 行，承载了 Chainlit UI 回调、RBAC 检查、文件上传、图层控制、管线调度等至少 6 种职责。`frontend_api.py` 有 3302 行，虽然做了委托（每个 handler 函数 lazy import 子模块），但 153 个路由函数堆在一个文件里仍然违背单一职责原则。

`Architecture_Review_and_Refactoring_Plan.md` 早就诊断了这个问题，提出了三阶段重构计划。但到今天，大部分重构还是 `- [ ]`（未完成）。为什么？因为 AI 太擅长往现有文件里"加东西"了——加一个 API 端点只要告诉 AI "在 frontend_api.py 里加一个 /api/xxx"，30 秒就完成了。拆分文件的收益是长期的，加功能的快感是即时的。

**（2）版本号膨胀**

7 周走了 v12.0 到 v18.5，平均每周一个大版本。每个版本都带着一串新功能：

- v16.0：4 个语义算子 + 13 个子 Agent + 5 种错误恢复策略 + YAML guardrails
- v17.0：时序对齐 + 语义增强 + 冲突解决 + 可解释性
- v18.5：NL2Workflow + 提示词自动优化 + 15 个评估器 + Palantir 风格 UI 重设计

这不是版本迭代，这是**功能堆积**。版本号在增长，但系统的内聚度并没有随之提高。功能之间的关系越来越像一张网而不是一棵树。

**（3）缺少代码质量基础设施**

项目没有配置 linter（没有 `.flake8`、`ruff.toml`、`pyproject.toml` 中的 lint 配置）。在 AI 生成代码的场景下，linter 其实比人工编码时更重要——因为 AI 每次生成的代码风格可能有微妙差异，需要 linter 来强制统一。

**（4）架构评审建议未执行**

`Architecture_Review_and_Refactoring_Plan.md` 提出了三条关键建议：

1. 拆分 `app.py`（迁移 Auth/API/UI 到独立模块）
2. 配置驱动化（Prompt 外置、Agent 模板化）
3. GIS 引擎抽象层（`GISEngine` 策略模式）

第 2 条部分完成了（Prompt 已迁移到 `prompts/` YAML），其他基本未动。这说明在 AI Coding 模式下，**结构性重构很容易被新功能挤掉优先级**。
## 2. AI Coding 如何放大了 ABCD 的不均衡

回头看这 47 天，一个清晰的模式浮现出来：

**D 的边际成本趋近于零，导致"再加一个 feature"几乎没有阻力。**

在没有 AI 的时代，写一个新功能需要：理解需求→设计接口→编码→写测试→调试→集成。每一步都有时间成本，这个成本本身就是一种"刹车机制"——它迫使你在动手之前多想一想："这个功能真的需要吗？它和已有的功能是什么关系？"

有了 Claude Code 之后，对话框里打一段话，30 秒后代码就生成了，测试也写好了。刹车消失了。这导致了三个后果：

### 2.1 Roadmap 膨胀陷阱

从 v12 到 v18.5 的 roadmap 本质上是一个不断膨胀的功能列表。每次看到一个新的竞品（SeerAI 的数据编排、DeerFlow 的工程质量、AgentArts 的平台能力、Datus.ai 的上下文工程），就在 roadmap 里加一批新条目。

为什么会这样？因为 AI 让"对标竞品"变得太容易了。以前看到竞品有一个功能，你会想"实现这个功能要两周，值不值"；现在你会想"让 AI 做大概两小时就行"。两小时的成本几乎不构成决策障碍。

但是——**加上去的功能需要有人维护**。更重要的是，**加上去的功能需要和已有的功能构成一个有内聚力的整体**。而这个"内聚力"正是 C-分析工作流应该提供的，它不会因为 AI 写代码快了就自动出现。

### 2.2 C 层的隐形代价

每次让 AI 加一个功能，AI 会在 D 层做**局部最优**——它会新建一张数据库表、新增一个 toolset、新增几个 API 端点。在局部看，每次生成的代码质量都不错：模式一致、有测试、有文档。

但 AI 不会自觉地问："这个新概念和已有的概念是什么关系？已有的 `agent_data_catalog` 表和你新建的 `agent_data_assets` 表是不是同一件事？"

这个问题只有 C-分析才能回答。而 C-分析需要人来做——需要有人画一张领域类图，看看新加的东西放在图上是否合理。

**AI 编码的速度越快，C 层缺失的代价积累得越快。**在传统开发中，两个开发者各自建了一张功能重叠的表，可能两周后在代码评审时发现；在 AI 编码中，两张重叠的表可能两小时内就都建好了，而且各自都有完整的 CRUD 和测试——看起来一切正常，直到你试图让两张表的数据联动时才发现矛盾。

这就是 TD-001 的真实故事。

### 2.3 "Spec-Driven"的幻觉

我的 CLAUDE.md 文件有 250 多行，详细记录了架构、模块、mock 目标、运行命令。这看起来像是 "Spec-Driven" 对吧？

但正如潘老师指出的："缺少方法学的知识，Spec 越来越像伪代码。"

CLAUDE.md 里写的是 "db_engine.py: Singleton SQLAlchemy engine"、"semantic_layer.py: 5-min TTL cache"——**这些全是 D 层的描述**。它告诉 AI "系统现在长什么样"，但不告诉 AI "系统的核心域概念是什么、概念之间什么关系、什么能做什么不能做"。

真正能约束 AI 行为的 Spec 应该是 C 层的：

- "数据资产（DataAsset）是系统的核心聚合根，包含四层元数据"
- "一个数据资产可以有多个版本（DataAssetVersion），版本之间有 derives_from 血缘关系"
- "质检缺陷（QCDefect）必须关联到一个数据资产，严重程度按 GB/T 24356 三级分类"

如果 CLAUDE.md 里有这样的领域约束，AI 在新增功能时就会受到 C 层的约束，而不是每次都做局部最优的 D 层决策。

## 3. 改进路径

### 3.1 补 C-分析的课：画一张核心域类图

**最优先、最重要的一步。**

具体做法：

1. 拿出一张白纸（或打开 EA / PlantUML），画出系统当前涉及的**核心域概念**：DataAsset、SpatialEntity、QCDefect、Workflow、Skill、Tool、User、Pipeline、FusionTask、CausalAnalysis
2. 标注它们之间的关系：聚合、组合、关联、泛化
3. 标注每个概念的关键属性（不是数据库字段，是领域属性）
4. 用这张图去审视现有的 52 个 migration：哪些表对应同一个概念？哪些表应该合并？哪些关系在数据库里缺失了？

这张图应该加入 CLAUDE.md，成为 AI 编码时的 C 层约束。以后每次新增功能，先在类图上看看新概念放在哪里，再让 AI 去实现。

### 3.2 统一领域模型的"真相源"

把 `semantic_layer.py` 缓存、`gis_ontology.yaml`、`agent_semantic_registry` 表这三个语义源统一到一个"领域模型包"。

建议方案：

- `gis_ontology.yaml` 作为**静态基线**（领域专家维护，版本化）
- `agent_semantic_registry` 表作为**动态扩展**（用户自定义语义，DB 存储）
- `semantic_layer.py` 作为**查询接口**（合并两个来源，提供统一 API）
- 在 CLAUDE.md 中明确写："语义的权威来源是 YAML + DB，代码中不允许硬编码语义等价关系"

### 3.3 把 B-需求从功能列表恢复到用例级别

停止在 `roadmap.md` 中混写需求和设计。具体做法：

1. `roadmap.md` 只保留版本规划和技术设计决策（D 层）
2. 新建 `docs/use_cases/` 目录，对每个核心功能写正式的**用例规约**：
   - 用例名（动宾结构，如"上传空间数据"、"执行缓冲区分析"）
   - 主执行者
   - 前置条件
   - 主成功场景（编号步骤）
   - 扩展场景（异常和替代路径）
   - 业务规则
   - 后置条件

3. 区分"提升销售"（B-需求层）和"降低成本"（D-设计层）的条目。例如：
   - "支持自然语言执行空间聚类分析"——这是 B（面向涉众的功能）
   - "使用 asyncpg 替代 psycopg2"——这是 D（降低成本的设计决策）

### 3.4 用 AI 做 ABC 而不是只做 D

潘老师在课件（第 39 页）给 AI 定义了两个角色：

- **角色 A（无所不能的顶级专家）**：在你不擅长的领域，AI 比你强
- **角色 B（十分聪明的学生）**：在你擅长的领域，你用方法学引领 AI

具体做法：

**A-业务建模阶段**：在开始新版本前，把目标组织的背景信息和业务流程丢给 AI，让它以"角色 A"身份帮你识别改进机会。就像在 `软件方法_定位目标组织规格.md` 中做的那样——但要在**编码之前**做，不是之后。

**B-需求阶段**：把已有的 PRD 和用例列表给 AI，让它以"角色 B"身份帮你推导新功能的用例步骤和业务规则。AI 可能会遗漏，但它至少能帮你搭出用例规约的框架，你再来审查和补充。

**C-分析阶段**：把核心域类图给 AI，让它在新增功能时先回答"这个新概念在类图上应该放在哪里"，再开始写代码。如果 AI 的回答和类图矛盾，先更新类图（人来判断），再让 AI 按新类图写代码。

关键是把这些制品（业务流程图、用例规约、核心域类图）放入 CLAUDE.md 或作为 AI 的上下文输入。**没有输入 ABC 约束的 AI，只会做 D。**

### 3.5 控制版本节奏

从"每周一个大版本"改为"每两周一个版本 + 版本内 C 层审查"。

每个版本结束时做一次**C 层一致性检查**：

- 这个版本新增了几张 DB 表？它们在领域类图上的位置是什么？
- 新增的 toolset 是按领域概念组织还是按技术活动组织？
- 新增的 API 端点和已有端点有没有语义重叠？
- CLAUDE.md 的架构描述和实际代码是否一致？

这个检查本身也可以用 AI 来辅助——让 AI 读取最新的 migration、toolset、API 列表，对照领域类图检查一致性。但最终的判断权在人手里。

## 4. 结语

### 4.1 一句话总结

**AI Coding 没有消除 A→B→C→D 的推理过程，它只是把 D 做快了。**

做快了 D 之后，ABC 的质量成为真正的瓶颈——不是因为 ABC 本来不重要，而是因为 D 的低成本让你可以不做 ABC 也能"出活"，直到系统复杂度超过了"朦朦胧胧在大脑里做 ABC"的承载极限。

### 4.2 这个项目在 ABCD 上的评分（自评）

| 工作流 | 评分 | 说明 |
|--------|------|------|
| A-业务建模 | **60/100** | 做了，但时间顺序倒了；有方法论意识但执行不够严格 |
| B-需求 | **70/100** | PRD 质量高且先于编码，但后续退化成功能列表 |
| C-分析 | **35/100** | 最薄弱；有零星的领域概念（ontology、taxonomy），但没有统一的领域模型 |
| D-设计/实现 | **80/100** | 速度、测试质量、模块基本面都不错，但有 God Module 和版本膨胀问题 |

ABCD 之间的不均衡一目了然：**D 最强，C 最弱，B 居中，A 有意识但不够**。这恰恰印证了潘老师的观察——软件方法学的关注顺序是 D→C→B→A，我们最先学会的是 D，最后才去思考 A。

### 4.3 潘老师的洞察验证

潘老师在书中写道：

> "也幸亏如此（需求和设计之间不存在刻板的一对一映射），否则人工智能现在就可以轻易学习其中的映射套路，取代软件开发人员完成所有的工作。目前，软件开发人员存在的价值之一就是根据自己掌握的领域知识和软件开发知识，努力在很多可能的映射方案中挑选出最合适的映射方案，以及创造出更好的映射方案。"

这段话在 2026 年的 AI Coding 实践中完全成立。Claude Code 可以帮我把一个需求映射到代码，但它不会帮我判断"这个映射方案是不是最优的"。判断映射方案的质量，需要 C-分析提供的核心域模型作为参照系。

**利润 ＝ 需求 － 设计。** AI 大幅压低了"设计"的成本，这是好事。但如果"需求"（以及支撑需求的业务建模和领域分析）没做好，利润依然是负的——你只是用更低的成本更快地做出了一个卖不出去的东西。
