# 传统数据平台智能体化策略分析

> 讨论日期：2026-04-04
> 背景：将传统时空数据治理平台（Vue + Java）智能体化的路径选择

---

## 1. 背景

现有系统是一个传统的时空数据治理平台（数据中台），支持 GIS 类时空数据和非空间数据的治理。

**系统构成**：
- 前端：Vue 框架
- 后台：Java
- 模块：数据标准、数据汇聚、数据质量、数据开发、数据服务、数据资产门户、系统管理、元数据

**目标**：将平台智能体化，首先从数据质量模块切入，基于智能体架构实现智能化的数据质检能力。

**核心问题**：如何将这类传统的软件代码以最佳方式智能体化？

---

## 2. 四种路径对比

### 路径 A：封装 RESTful API（OpenAPI）

**做法**：把现有 Java 后台的 REST 接口直接注册为 Agent 的 Tool

```
现有 Java API  →  OpenAPI Spec  →  Tool Definition  →  Agent 调用
```

**优势**：
- **最小改动、最快落地**（天级别）
- 复用全部已有业务逻辑和数据校验
- OpenAPI 3.0 规范可以半自动转成 Tool schema（参数名、类型、描述都有）
- 不动 Java 代码，风险最低

**劣势**：
- API 粒度问题——传统 API 是给前端 CRUD 设计的，不是给 Agent 设计的。例如"创建质检任务"可能需要先调 3 个接口拿参数再调 1 个接口提交，Agent 体验很差
- 缺乏语义描述——API 的 `description` 通常写给开发者看，不是写给 LLM 看的
- 错误处理不友好——HTTP 500 + Java stacktrace 对 LLM 来说是噪音

**适合场景**：功能已经稳定、API 粒度刚好匹配 Agent 单步操作的模块

---

### 路径 B：封装 MCP Server

**做法**：在 Java 服务外层包一个 MCP Server（Python/TS），内部调用现有 API 或直连数据库

```
Agent  ←MCP协议→  MCP Server (Python/TS)  →  Java REST API / 直连DB
```

**优势**：
- MCP 是 Agent 生态的事实标准，工具链丰富
- **可以重新设计工具粒度**——把 3 个细碎 API 合并成 1 个语义完整的 Tool
- 支持 stdio / SSE / streamable-http 三种传输，部署灵活
- 可以加入 Agent 友好的 description 和 error message

**劣势**：
- 多了一层中间服务需要维护
- 如果只是简单转发 API，本质上和路径 A 没区别
- MCP Server 本身需要处理认证透传

**适合场景**：需要重组工具粒度、或需要跨多个 Java 微服务编排的场景

---

### 路径 C：封装 CLI

**做法**：通过命令行工具调用现有功能

**结论：不推荐。** CLI 的文本解析脆弱、错误处理差、结构化输出难保证。除非你的系统本身就是 CLI 优先的（如 gdal、ogr2ogr），否则没有理由选这条路。

---

### 路径 D：AI Native 重写

**做法**：基于 Agent 框架（ADK / LangGraph / AutoGen 等）从零构建

```
Agent Framework  →  LLM Agent + Tools (直接操作DB/引擎)  →  新前端
```

**优势**：
- **体验最好**——工具粒度、错误处理、上下文传递都为 Agent 设计
- 可以引入 Agent 独有能力：语义路由、多 Agent 协作、自我纠错循环、记忆
- 不受遗留架构限制

**劣势**：
- **成本最高**——需要用 Python/TS 重写业务逻辑
- 你的数据质量规则、标准体系、工作流引擎都沉淀在 Java 里，重写等于重做
- 风险高，周期长（月级别）

**适合场景**：全新模块、或现有模块逻辑简单但交互需要彻底改变

---

## 3. 路径对比总结表

| 维度 | A. OpenAPI 封装 | B. MCP Server | C. CLI | D. AI Native |
|------|----------------|---------------|--------|-------------|
| 落地速度 | 天级 | 周级 | 周级 | 月级 |
| 改动量 | 最小 | 中等 | 中等 | 最大 |
| 工具粒度控制 | 差（受限于现有API） | 好（可重组） | 差 | 最好 |
| Agent 体验 | 一般 | 好 | 差 | 最好 |
| 维护成本 | 低 | 中 | 高 | 高 |
| 风险 | 最低 | 低 | 中 | 高 |
| 推荐度 | Phase 1 | Phase 2 | 不推荐 | Phase 3 |

---

## 4. 推荐策略：混合渐进（Hybrid Incremental）

**最佳方案不是四选一，而是分层组合：**

```
┌─────────────────────────────────────────────────┐
│  AI Native 层 (新增)                              │
│  ┌──────────────────────────────────────────┐    │
│  │  Agent Orchestrator (ADK/LangGraph)      │    │
│  │  - 语义意图路由                            │    │
│  │  - 多 Agent Pipeline (质检→分析→修正)      │    │
│  │  - 自我纠错 Loop                          │    │
│  │  - 质检报告生成 (LLM Native)              │    │
│  └──────┬──────────────┬──────────────┬─────┘    │
│         │              │              │          │
│  ┌──────▼─────┐ ┌──────▼──────┐ ┌────▼───────┐  │
│  │ MCP Server │ │ OpenAPI Tool│ │ Native Tool│  │
│  │ (重组粒度) │ │  (直接复用)  │ │  (新写)    │  │
│  └──────┬─────┘ └──────┬──────┘ └────┬───────┘  │
│         │              │              │          │
├─────────▼──────────────▼──────────────▼──────────┤
│  现有 Java 平台层                                  │
│  数据质量引擎 / 元数据服务 / 标准库 / 工作流引擎     │
└─────────────────────────────────────────────────┘
```

---

### Phase 1：直接封装 OpenAPI（1-2 周）

把现有 Java 质检 API 直接注册为 Tool，快速验证 Agent 交互模式：

- `run_quality_check(dataset_id, rule_set)` → 调现有接口
- `get_quality_report(task_id)` → 调现有接口
- `list_quality_rules()` → 调现有接口

**目标**：快速跑通端到端，验证 LLM 能不能正确理解和调用你的质检工具。

---

### Phase 2：MCP 重组关键工具（2-4 周）

发现 API 粒度不合适的地方，用 MCP Server 重新封装：

- 把"查询规则 + 选择规则 + 执行检查 + 获取结果"合并成一个 `smart_quality_check(dataset, description)` 工具
- 加入 Agent 友好的错误信息和结果摘要
- 质检规则的自然语言描述（给 LLM 看的元数据）

---

### Phase 3：AI Native 新增增值能力（4-8 周）

用 Agent 框架原生实现**传统系统做不了的事**：

- **智能质检规则推荐**：根据数据 profile 自动推荐适用的质检规则（LLM 推理）
- **异常归因分析**：发现质量问题后，用 LLM 分析可能的根因
- **自动修正建议**：生成修复 SQL/脚本，人工确认后执行
- **质检报告生成**：自然语言质检报告（不是模板填充）
- **多 Agent 质检流水线**：探索→检测→分析→修正→复核

---

## 5. 核心原则

| 原则 | 说明 |
|------|------|
| **不重写已沉淀的业务逻辑** | Java 里的质检规则引擎、标准体系是核心资产，封装复用 |
| **只重写 Agent 独有能力** | 语义理解、智能推荐、自然语言交互——这些是传统系统做不了的 |
| **工具粒度为 Agent 设计** | 一个 Tool 对应一个完整的语义动作，不是一个 CRUD 接口 |
| **渐进式，快速验证** | 先跑通再优化，不要一步到位 |
