# Harness Engineering 实践指南

> 基于 GIS Data Agent (ADK Edition) 项目开发经验总结
> 2026-03-30

## 什么是 Harness Engineering

"如何高效驾驭 AI 编程助手"的工程能力。不是提示词技巧，而是一套与 AI 协作的工程方法论。

---

## 核心框架

```
1. 诊断先于修复  — 先让 AI 读代码、建立理解、输出分析
2. 测试锚定目标  — 用失败测试定义"什么叫修好了"
3. 分而治之      — 后端和前端分开，一次只改一层
4. 短反馈循环    — 让 AI 自己跑测试验证
5. 指令要具体    — 指定文件、函数、预期行为
6. 对话要专注    — 一个对话一个目标，不要发散
```

---

## 原则详解

### 1. 诊断先于修复

很多时候我们的对话模式是：
```
"工作流有 bug，帮我修" → AI 开始猜测并改代码 → 改错了 → 再改 → 循环
```

更高效的模式：
```
"先读这几个文件，分析工作流执行路径，告诉我你理解的数据流"
→ AI 输出理解 → 你纠正
→ "现在复现这个 bug 的最小测试用例是什么？"
→ 有了精确定位再修
```

**关键**：让 AI 先输出它对问题的理解，你来判断理解是否正确。如果理解就错了，修出来的代码一定是错的。

### 2. 测试锚定目标

修 bug 最怕的是"感觉修好了但其实没有"。用测试把 AI 的目标从模糊的"修 bug"变成精确的"让这个测试通过"。

```
# 先写一个失败的测试
"写一个 test 复现这个 bug：当工作流步骤 A 完成后，步骤 B 拿不到 A 的 context"

# 然后让 AI 修到这个测试通过
"修 workflow_engine.py 让这个测试通过，不要改测试"
```

这就是 TDD 式的 harness — 测试就是需求的精确表达。

### 3. 分而治之

当一个 bug 横跨多个文件（比如 workflow_engine.py + workflow_routes.py + WorkflowsTab.tsx）：

- 先用一个对话做**纯诊断** — 只读代码、跑测试、分析日志，产出一份明确的 bug 根因报告
- 再用一个新对话做**精准修复** — 把根因报告贴进去，只动确定要改的文件
- 后端和前端分开修 — 先确认 API 返回正确，再改前端

**一个对话承载过多内容时，context window 被大量文件填满，AI 的注意力会分散。**

### 4. 短反馈循环

如果 AI 改了代码 → 你手动测试 → 发现不对 → 再描述问题 → AI 再改... 这个循环太慢。

更好的方式：
```
"改完后运行 pytest data_agent/test_workflow_engine.py -v，把结果给我看"
```

让 AI 自己跑测试、看结果、迭代，把反馈循环缩短到秒级。

### 5. 指令要具体

| 粗粒度（低效） | 细粒度（高效） |
|---|---|
| "修复工作流 bug" | "读 workflow_engine.py 的 `execute_dag` 方法，解释 step context 是如何在节点间传递的" |
| "前后端都要改" | "先只改后端，确认 API 返回正确后再改前端" |
| "参考之前的实现" | "参考 pipeline_runner.py 的 PipelineResult 模式，保持一致" |

### 6. 对话要专注

一个对话一个目标，不要发散。常见反模式：

- 修着 bug 顺手重构了相关代码
- 一个对话里同时修 3 个不相关的问题
- 修完 bug 又加新功能

---

## 常见反模式与修正

| 反模式 | 问题 | 修正 |
|---|---|---|
| 一上来就说"帮我修" | AI 缺乏上下文，靠猜 | 先让 AI 读代码建立理解 |
| 改了又改，循环多轮 | 没有明确的"修好"标准 | 先写失败测试，以测试通过为目标 |
| 一次对话改太多文件 | context 被稀释 | 分阶段：诊断→后端→前端 |
| 手动验证后口头描述结果 | 信息丢失、描述不精确 | 让 AI 自己跑命令看输出 |
| 说"有个 bug"不给细节 | AI 只能猜 | 给出具体症状、错误信息、复现步骤 |
| 同时修 bug + 加功能 | 目标混乱 | 严格分离，一次只做一件事 |

## 实战 Checklist

遇到 bug 时，按顺序执行：

- [ ] **描述症状**：具体的错误信息、日志、复现步骤
- [ ] **让 AI 读相关代码**：指定具体文件和函数
- [ ] **让 AI 输出理解**：数据流、调用链、状态变化
- [ ] **确认理解正确**：你来判断 AI 的分析是否准确
- [ ] **写最小复现测试**：用测试精确定义 bug
- [ ] **分层修复**：后端先行，前端跟进
- [ ] **AI 自验证**：每次改动后让 AI 跑测试
- [ ] **回归检查**：跑完整测试套件确认没有副作用

---

## 适用场景

这套方法论不仅适用于 debug，也适用于功能开发：

- **新功能**：先让 AI 分析现有架构 → 输出设计方案 → 你审核 → 再实现
- **重构**：先让 AI 画出依赖关系 → 确认影响范围 → 分步重构
- **性能优化**：先让 AI profiling → 定位瓶颈 → 针对性优化

核心思想始终一致：**理解先于行动，精确先于广泛，验证先于交付。**
