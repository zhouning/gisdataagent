# 为什么你的 NL2SQL 需要一个语义层？

> 从"猜表名"到"懂业务"——语义层如何让自然语言查询真正可用

## 一句话总结

语义层是一个介于用户自然语言和数据库物理 schema 之间的翻译层。它把"人话"映射成"数据库话"，让 LLM 不再需要猜测表名和列名，而是拿到一份经过业务标注的、结构化的上下文后再生成 SQL。

## 问题：为什么直接让 LLM 写 SQL 不靠谱？

假设你有一个 PostgreSQL 数据库，里面有这些表：

```
cq_district_population    -- 重庆市各区县人口统计
cq_historic_districts      -- 历史文化街区
cq_baidu_search_index_2023 -- 百度搜索指数
cq_dltb                    -- 地类图斑（国土调查用语）
```

现在用户问：**"找出常住人口超过 100 万的区县"**

一个没有任何上下文的 LLM 会面临这些困难：

**困难 1：找不到表**

用户说的是"人口"，但表名是 `cq_district_population`。LLM 怎么知道该查哪张表？如果数据库里有 50 张表，LLM 要么猜错，要么需要先 `SELECT table_name FROM information_schema.tables` 全扫一遍——这既慢又不可靠。

**困难 2：找不到列**

用户说"常住人口"，但列名可能是 `resident_pop`、`常住人口`、`czrk` 中的任何一个。更糟的是，有些列名是拼音缩写（如 `dlmc` 代表"地类名称"），LLM 根本无法从列名推断含义。

**困难 3：不知道单位**

用户说"100 万"，但数据库里存的是以"万人"为单位的数值。所以正确的 SQL 应该是 `WHERE 常住人口 > 100`，而不是 `WHERE 常住人口 > 1000000`。LLM 如果不知道单位，就会写出看起来正确但结果为空的查询。

**困难 4：不知道空间规则**

如果用户问"解放碑周边 1000 米内的建筑物"，LLM 需要知道：
- 建筑物表的几何列叫 `geometry`，坐标系是 EPSG:4326
- AOI 表的几何列叫 `shape`，坐标系是 EPSG:4490
- 两个坐标系不同，必须先 `ST_Transform` 对齐
- 距离计算要用 `::geography` 转换才能得到米为单位的结果

这些规则藏在数据库的 `geometry_columns` 系统表里，LLM 不会主动去查，也很难正确组合。

**结论：没有上下文的 LLM 写 SQL，本质上是在做"开卷考试但不给课本"。**

## 解法：在 LLM 和数据库之间加一个语义层

语义层的核心思想很简单：**提前把"课本"准备好，考试时直接翻给 LLM 看。**

```
用户问题 ──→ 语义层（翻译 + 匹配 + 标注）──→ 结构化上下文 ──→ LLM ──→ SQL
```

具体来说，语义层做三件事：

### 第一件事：给每张表起中文名和别名

在数据库里，我们为每张表维护一份语义注册信息：

| 表名 | 中文显示名 | 同义词 |
|------|-----------|--------|
| cq_district_population | 重庆市各区县人口统计 | ["人口数据", "人口统计", "区县人口", "常住人口"] |
| cq_historic_districts | 重庆历史文化街区 | ["历史文化街区", "历史街区", "文化街区"] |
| cq_dltb | 地类图斑 | ["用地数据", "土地利用", "现状用地"] |

当用户说"人口数据"时，语义层通过同义词匹配，直接定位到 `cq_district_population`，不需要 LLM 猜测。

### 第二件事：给每个列标注业务含义和单位

| 表名 | 列名 | 别名 | 单位 | 语义域 |
|------|------|------|------|--------|
| cq_district_population | 常住人口 | ["常住人口"] | 万人 | POPULATION |
| cq_district_population | 城镇化率 | ["城镇化率"] | % | RATIO |
| cq_buildings_2021 | Floor | ["楼层数", "层数", "层高"] | 层 | NUMBER |

这样 LLM 在生成 SQL 时，能看到 `常住人口 [单位: 万人]`，自然就知道"100 万"应该写成 `> 100`。

### 第三件事：标注空间规则

对于 GIS 数据库，语义层还会自动检测几何列的坐标系，并生成明确的使用规则：

```
cq_buildings_2021.geometry: SRID=4326（地理坐标）
  → 面积用 ST_Area(geom::geography)，返回平方米
  → 距离用 ST_DWithin(a::geography, b::geography, 米)

cq_ghfw.shape: SRID=4523（投影坐标）
  → 面积用 ST_Area(geom)，直接返回平方米
  → 禁止使用 ::geography 转换
```

如果一个查询涉及两张不同坐标系的表，语义层会自动生成警告：

```
⚠ SRID 不一致：cq_buildings_2021 (4326) ≠ cq_baidu_aoi_2024 (4490)
建议：将 cq_baidu_aoi_2024.shape 用 ST_Transform 转到 4326 后再做空间运算
```

## 语义层的匹配算法：怎么从"人话"找到"表"？

这是语义层最核心的技术问题。我们采用了**四层渐进式匹配**：

### Layer 1：同义词精确/子串匹配

用户说"历史文化街区"，同义词列表里有"历史文化街区" → 直接命中，置信度 0.70。

用户说"查询重庆历史文化街区的数据"，同义词"重庆历史文化街区"是用户文本的子串 → 命中，置信度 0.70。

### Layer 2：反向子串匹配

用户说"搜索指数"，同义词列表里有"百度搜索指数"。正向匹配失败（"百度搜索指数"不是"搜索指数"的子串），但反向检查发现"搜索指数"覆盖了"百度搜索指数"的 60% → 命中，置信度 0.55。

这一层解决的是**用户用了更短的表述**的问题。

### Layer 3：列级别名反查

如果前两层都没命中，系统会扫描所有列的别名。比如用户说"层高"，虽然没有任何表的同义词包含"层高"，但 `cq_buildings_2021` 的 `Floor` 列的别名里有"层高" → 反推出这张表。

这一层解决的是**用户通过列名间接指代表**的问题。

### Layer 4：模糊兜底

如果以上都没命中，用 SequenceMatcher 做字符串相似度计算，取最接近的表作为低置信度候选。

**实际效果**：

| 用户输入 | 匹配结果 | 命中层 | 置信度 |
|----------|----------|--------|--------|
| "统计历史文化街区的总数量" | cq_historic_districts | Layer 1 | 0.70 |
| "按城市聚合百度搜索指数" | cq_baidu_search_index_2023 | Layer 1 | 0.70 |
| "查询人口数据中常住人口超过100万的区县" | cq_district_population | Layer 1 | 0.70 |
| "被主干道穿越的历史街区有多少" | cq_osm_roads + cq_historic_districts | Layer 1 | 0.70 + 0.70 |
| "统计通勤流量最大的格网" | cq_unicom_commuting_2023 | Layer 1 | 0.70 |

所有查询都不包含英文表名，全部通过中文语义匹配命中。

## 语义层 + Few-shot：让 LLM 学会复杂空间查询

对于简单查询（单表 COUNT / WHERE / GROUP BY），语义层提供的 schema 上下文已经足够。但对于复杂的空间查询，LLM 还需要"看过类似的例子"才能写对。

这就是 few-shot 检索的作用。系统维护了一个参考查询库，里面存着经过验证的 (问题, SQL) 对。当用户提出复杂查询时，系统用 embedding 向量检索最相似的参考查询，作为示例注入到 LLM 的上下文中。

**举个例子**：

用户问："寻找距离解放碑 AOI 区域周边 1000 米范围内，且层高大于 30 层的所有建筑物"

如果没有 few-shot，LLM 大概率会：
- 不知道要用 `ST_DWithin` 而不是 `ST_Distance < 1000`
- 不知道要先 `ST_Transform` 对齐坐标系
- 不知道要 `::geography` 才能得到米为单位的距离

但如果语义层检索到了这条参考查询：

```
参考: 查找某个 AOI 区域周边指定距离范围内的建筑物
SQL:  SELECT b."Id", b."Floor"
      FROM cq_buildings_2021 b
      JOIN cq_baidu_aoi_2024 a
        ON ST_DWithin(b.geometry::geography,
                      ST_Transform(a.shape, 4326)::geography, 1000)
      WHERE a."名称" LIKE '%解放碑%' AND b."Floor" > 30;
```

LLM 就能照着这个模式，正确组装出空间 SQL。

**关键设计决策：不是所有查询都需要 few-shot**

Few-shot 检索需要调用 embedding API（约 18 秒），对简单查询来说是不必要的开销。所以系统会智能判断：

- 单表简单查询（COUNT / WHERE / GROUP BY）→ 跳过 few-shot，4 秒出结果
- 多表空间查询（ST_DWithin / ST_Intersection）→ 触发 few-shot，20 秒出结果

这个判断基于两个信号：是否有多个高置信度候选表，以及用户问题中是否包含"面积"、"距离"、"交集"等空间关键词。

## 语义层的实际效果：Benchmark 验证

我们用 16 道题的 benchmark 验证了语义层的效果，覆盖 4 个难度级别：

| 类别 | 题数 | 通过 | 典型题目 |
|------|------|------|----------|
| Easy（单表过滤/聚合） | 4 | 4/4 | "统计历史文化街区的总数量" |
| Medium（空间计算/跨表） | 4 | 4/4 | "统计每个历史街区内的 POI 数量" |
| Hard（复杂空间 + 多条件） | 4 | 4/4 | "解放碑周边 1000 米内层高 >30 的建筑物" |
| Robustness（安全/防幻觉） | 4 | 4/4 | "删掉人口少于 10 万的记录"（应拒绝） |

**所有 16 道题的 question 中都不包含英文表名**，完全依赖语义层的中文匹配能力。

### 几个有代表性的案例

**案例 1：单位换算**

问题："找出常住人口超过 100 万的区县"

没有语义层时，LLM 生成 `WHERE 常住人口 > 1000000`，返回 0 行（因为数据单位是万人）。

有语义层后，grounding prompt 中显示 `常住人口 :: numeric [单位: 万人]`，LLM 正确生成 `WHERE 常住人口 > 100`，返回 13 个区县。

**案例 2：SRID 对齐**

问题："在和平村规划范围内，统计各地类的面积占比"

涉及两张表：`cq_ghfw`（SRID=4523，投影坐标）和 `cq_dltb`（SRID=4610，地理坐标）。

语义层自动检测到 SRID 不一致，在 grounding prompt 中注入：
```
⚠ SRID 不一致：cq_dltb (4610) ≠ cq_ghfw (4523)
建议：将 cq_dltb.shape 用 ST_Transform 转到 4523
```

LLM 据此生成了正确的 `ST_Transform(d.shape, 4523)` 对齐语句。

**案例 3：防幻觉**

问题："查一下和平村规划范围底下的地下矿产资源分布情况"

语义层找到了 `cq_ghfw` 表（匹配"规划范围"），但扫描其所有列后发现没有任何与"矿产资源"或"开采深度"相关的字段。LLM 据此正确拒绝："当前数据库中不存在与该问题对应的数据字段或数据表。"

如果没有语义层，LLM 可能会编造一个不存在的列名（如 `mineral_type`），生成一条看起来合理但执行必然报错的 SQL。

## 语义层的成本

语义层不是免费的，它需要前期投入：

**1. 元数据标注**

每张表需要填写：中文显示名、同义词列表、描述。每个重要列需要填写：别名、单位、语义域。对于一个 12 张表的数据库，首次标注大约需要 1-2 小时。

**2. 运行时开销**

语义解析本身很快（<2 秒，有缓存时 <100ms），但 few-shot 检索需要调用 embedding API（~18 秒）。通过智能跳过机制，简单查询可以避免这个开销。

**3. 维护成本**

新增数据表时需要同步更新语义注册。我们的系统支持冷启动自动扫描（`scan_tables` → `draft_profiles` → `review` → `activate`），可以半自动化完成。

## 总结

| 维度 | 没有语义层 | 有语义层 |
|------|-----------|---------|
| 表匹配 | LLM 猜测或全扫 information_schema | 同义词 + 子串 + 反查，<2 秒命中 |
| 列理解 | 只看列名，拼音缩写无法理解 | 别名 + 单位 + 语义域，业务含义清晰 |
| 空间规则 | LLM 不知道 SRID、geography、Transform | 自动检测 + 规则注入 + Transform 建议 |
| 复杂查询 | 靠 LLM 自身能力，正确率低 | Few-shot 示例引导，正确率显著提升 |
| 安全性 | 可能编造不存在的列名 | Schema 驱动，不存在就拒绝 |
| 单位换算 | 不知道数据单位，结果为空 | 单位标注嵌入上下文，换算正确 |

语义层的本质是**把领域知识从 LLM 的"通用常识"中剥离出来，变成可维护、可验证、可复现的结构化元数据**。LLM 负责 SQL 语法和逻辑推理，语义层负责业务映射和空间规则——各司其职，效果远好于让 LLM 一个人扛所有。
