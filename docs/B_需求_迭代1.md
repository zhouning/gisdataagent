## 1. 系统边界

**系统名称**：GIS Data Agent（时空数据智能体平台）

**定位**：时空数据治理平台的智能化入口，覆盖数据治理完整流程（步骤 1-6）。底层调用已有的时空数据治理平台能力，对操作人员呈现为一个统一的系统。

**与已有系统的关系**：

```
┌─────────────────────────────────────────────────┐
│              GIS Data Agent（智能化入口）           │
│                                                   │
│  步骤 1-4（AI 新增能力）    步骤 5-6（调用底座）     │
│  数据感知 / 标准对照 /      汇聚 / 质检 / 开发 /    │
│  模型推荐                   服务 / 资产 / 报告      │
│                                                   │
├─────────────────────────────────────────────────┤
│           时空数据治理平台（已有底座）                │
│  元数据管理 / 数据汇聚 / 数据质量 / 数据开发 /      │
│  数据服务 / 数据资产 / BI 可视化                    │
└─────────────────────────────────────────────────┘
```

## 2. 执行者（Actor）

| 执行者 | 类型 | 说明 |
|--------|------|------|
| **数据治理操作人员** | 业务工人 | 超图实施人员 + 客户方数据管理人员 + 客户方委托的第三方实施人员，统一为一类（使用相同功能） |
| **数据消费方** | 业务实体 | 行业智能体 / 传统应用系统，从 GIS Data Agent 获取治理后的数据 |
| **数据源系统** | 业务实体 | 被 GIS Data Agent 连接和扫描的外部数据系统（数据库、文件服务器、服务接口） |
| **时空数据治理平台** | 业务实体 | 底座，被 GIS Data Agent 调用执行治理操作 |
| **EA 在线仓库** | 业务实体 | 存放通用自然资源数据模型，被读取用于模型推荐 |
| **数据标准库** | 业务实体 | 存放已梳理的数据标准文件，在时空数据治理平台的数据标准模块中管理 |
| **时间** | 业务实体 | 触发定时任务（数据质量变化感知） |

## 3. 用例清单

### 3.1 完整用例列表

| 编号 | 用例名 | 主执行者 | 对应业务流程步骤 | 简述 |
|------|--------|---------|----------------|------|
| UC-01 | 接入数据源 | 数据治理操作人员 | 步骤 1 | 连接注册各类数据源（数据库、文件目录、服务接口），掌握数据源情况 |
| UC-02 | 生成数据画像 | 数据治理操作人员 | 步骤 2 | 对已接入数据自动体检：字段分布、空值率、坐标系、几何类型、数据量、质量评分 |
| UC-03 | 管理数据标准 | 数据治理操作人员 | 步骤 3 前置 | 导入、维护、智能化解析数据标准文件（字段规范、值域规则、拓扑规则等） |
| UC-04 | 执行标准对照分析 | 数据治理操作人员 | 步骤 3 | 数据画像与数据标准自动语义匹配，生成差距报告（不合规项、缺失项、需转换项） |
| UC-05 | 推荐数据模型调整方案 | 数据治理操作人员 | 步骤 4 | 基于差距报告 + EA 仓库中的通用数据模型，推荐项目数据模型的修改建议 |
| UC-06 | 执行数据治理 | 数据治理操作人员 | 步骤 5 | 调用时空数据治理平台执行数据汇聚、质检、开发、服务发布、资产登记 |
| UC-07 | 生成治理成果报告 | 数据治理操作人员 | 步骤 6 | 自动生成可验收报告：治理前后对比、质量指标变化、处理记录明细 |
| UC-08 | 监控数据质量变化 | 时间 | 持续运营 | 数据源变化时自动感知质量异常并预警 |
| UC-09 | 提供数据服务 | 数据消费方 | 对外输出 | 向行业智能体或传统应用提供治理后的数据访问能力 |
| UC-10 | 可视化数据分析 | 数据治理操作人员 | 辅助决策 | 图表 + 地图交互 BI + ChatBI，展示数据现状和治理成果 |

### 3.2 用例优先级分层

基于愿景（让山西总工看到"能出活、能验收"），用例按面向对象分三层：

#### 第一层：对决策者可见（P0 — 首批验证）

目的：**打动总工，证明系统能帮项目产出可验收成果。**

| 用例 | 总工看到的价值 |
|------|-------------|
| UC-05 推荐数据模型调整方案 | "这个系统懂业务，能给出专业的模型建议" |
| UC-06 执行数据治理 | "建议完了能直接干活，不是纸上谈兵" |
| UC-07 生成治理成果报告 | "有白纸黑字的验收材料，我能交差" |
| UC-10 可视化数据分析 | "治理前后的对比我一眼就能看懂" |

**验证场景**：在山西项目中选一个具体的数据治理子场景（如某一类自然资源数据的质检+治理），用这 4 个用例走完从"推荐方案"到"出验收报告"的闭环。

#### 第二层：对操作者可见（P1 — 提效落地）

目的：**解决实施人员"做不动"的根因，让能力一般的人也能完成步骤 1-4。**

| 用例 | 实施人员得到的价值 |
|------|----------------|
| UC-01 接入数据源 | 不用手动整理文件，系统自动扫描识别 |
| UC-02 生成数据画像 | 不用人肉看数据，系统自动出体检报告 |
| UC-03 管理数据标准 | 标准文件智能解析，不用逐条人工录入 |
| UC-04 执行标准对照分析 | 不用人脑逐项比对，系统自动匹配出差距 |

**这一层是项目能持续跑下去的保障**：第一层让总工点头采用，第二层让实施团队真正提效。没有第二层，第一层的演示效果无法在日常工作中复现。

#### 第三层：长期运营价值（P2 — 后续迭代）

目的：**从"项目交付"走向"持续运营"，体现长期价值。**

| 用例 | 长期价值 |
|------|---------|
| UC-08 监控数据质量变化 | 数据治理不是一次性的，治理后的数据也会劣化，需要持续感知 |
| UC-09 提供数据服务 | 治理好的数据能被其他系统使用，数据资产才真正产生价值 |

## 4. 关键用例规约（P0 首批 — 概要级）

以下对 P0 的 4 个用例做概要级描述。后续迭代将补充完整的用例步骤、扩展场景和业务规则。

### UC-05 推荐数据模型调整方案

| 项 | 内容 |
|---|------|
| 用例名 | 推荐数据模型调整方案 |
| 主执行者 | 数据治理操作人员 |
| 前置条件 | 已完成数据画像（UC-02）或人工提供了数据现状描述；已有通用数据模型（EA 仓库） |
| 主成功场景 | 1. 操作人员选择目标数据范围和适用的通用数据模型<br>2. 系统读取 EA 仓库中的通用模型结构<br>3. 系统对比数据现状（画像）与通用模型的差异<br>4. 系统结合数据标准要求，生成模型调整建议清单（新增字段/修改类型/删除冗余/调整关系）<br>5. 操作人员审查并确认调整方案<br>6. 系统将确认后的模型更新回 EA 仓库（或直接导入时空数据治理平台） |
| 业务规则 | 推荐方案必须基于已梳理的通用自然资源数据模型，不能凭空生成；操作人员有最终决定权 |
| 后置条件 | 产出一份可用于导入时空数据治理平台的项目数据模型 |

### UC-06 执行数据治理

| 项 | 内容 |
|---|------|
| 用例名 | 执行数据治理 |
| 主执行者 | 数据治理操作人员 |
| 前置条件 | 项目数据模型已导入时空数据治理平台；数据源已接入 |
| 主成功场景 | 1. 操作人员描述治理目标（自然语言或选择预设流程）<br>2. 系统根据数据模型和数据现状，推荐治理流程（汇聚→质检→开发→服务→资产的具体步骤和参数）<br>3. 操作人员确认或调整治理流程<br>4. 系统调用时空数据治理平台逐步执行<br>5. 系统实时反馈执行进度和中间结果<br>6. 治理完成，系统汇总执行结果 |
| 业务规则 | 治理流程的每一步都可由操作人员干预和调整；质检规则来源于数据标准 |
| 后置条件 | 数据完成治理，质量指标达到预设标准 |

### UC-07 生成治理成果报告

| 项 | 内容 |
|---|------|
| 用例名 | 生成治理成果报告 |
| 主执行者 | 数据治理操作人员 |
| 前置条件 | 至少一轮数据治理已执行完成 |
| 主成功场景 | 1. 操作人员选择报告范围（数据集、时间段、治理批次）<br>2. 系统自动汇总：治理前数据质量 vs 治理后数据质量<br>3. 系统生成报告内容：质量指标对比、处理记录明细、异常项清单、治理建议<br>4. 操作人员预览并编辑报告<br>5. 系统导出为可交付格式（Word/PDF） |
| 业务规则 | 报告格式需满足项目验收要求；质量指标需可追溯到具体数据记录 |
| 后置条件 | 产出一份可用于项目验收的治理成果报告 |

### UC-10 可视化数据分析

| 项 | 内容 |
|---|------|
| 用例名 | 可视化数据分析 |
| 主执行者 | 数据治理操作人员 |
| 前置条件 | 已有数据画像或治理成果数据 |
| 主成功场景 | 1. 操作人员通过自然语言描述想看的内容（ChatBI）或选择预设看板<br>2. 系统生成图表 + 地图交互可视化<br>3. 操作人员可钻取、筛选、切换维度<br>4. 操作人员可将可视化结果嵌入治理成果报告 |
| 业务规则 | 空间数据必须支持地图渲染；支持治理前后的对比视图 |
| 后置条件 | 操作人员获得数据的可视化洞察 |

## 5. 与原 PRD 的对照分析

| 维度 | 原 PRD (2026-02-22) | B-需求迭代 1 (本文档) |
|------|---------------------|---------------------|
| **系统定位** | 独立的 AI 地理空间分析平台 | 时空数据治理平台的智能化入口 |
| **主执行者** | 非 GIS 背景的数据分析师 | 数据治理操作人员（超图实施 + 客户方 + 第三方） |
| **核心场景** | 上传 Excel → 自然语言空间分析 | 拿到数据 → 搞清现状 → 对照标准 → 调整模型 → 执行治理 → 出报告 |
| **用例来源** | 从想象的用户画像推导 | 从真实的山西项目痛点推导 |
| **优先级依据** | P0/P1/P2 按功能重要性分 | 按决策者可见 / 操作者提效 / 长期运营分层 |
| **验证方式** | 假想的演示场景 | 山西项目的一个具体数据治理子场景 |

### 原 PRD 中的哪些内容仍然有效？

- **Problem Statement**（空间数据分析断层）— 有效，但需要调整：断层不是"非 GIS 用户不会分析"，而是"实施人员不会做数据治理的前置步骤"
- **Non-Goals 的思路**（明确不做什么）— 有效，B 迭代 2 需要重新定义 Non-Goals
- **成功度量指标的框架**（领先+滞后指标）— 有效，但指标内容需要替换为"单场景交付周期"等

### 原 PRD 中需要修正的内容？

- 4 个用户画像需要替换为基于实际项目角色的执行者定义
- 12 个用户故事需要替换为从业务流程推导出的用例
- 功能列表需要按用例优先级重新组织
