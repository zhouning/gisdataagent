## 1. D 改进的核心原则

**GIS Data Agent 不是重建时空数据治理平台，而是在已有底座之上加"知识管理层 + 智能交互层"。**

21 个核心域概念的"管理"能力大部分已在时空数据治理平台中实现。GIS Data Agent 的价值是：

1. 用 AI 让操作人员不需要手动操作底座的每一个功能模块
2. 补充底座缺失的能力（差距报告、治理成果报告、ChatBI）
3. 将 40 个项目积累的领域知识结构化沉淀为 AI 可用的知识库

## 2. 目标架构（三层）

```
┌──────────────────────────────────────────────┐
│         GIS Data Agent                        │
│                                              │
│  ┌────────────────────────────────────────┐  │
│  │   智能交互层（面向操作人员的 AI 入口）    │  │
│  │  · 自然语言对话驱动治理流程              │  │
│  │  · AI 推荐（模型调整/治理流程/算子选择） │  │
│  │  · 治理成果报告自动生成                  │  │
│  │  · ChatBI 可视化                        │  │
│  ├────────────────────────────────────────┤  │
│  │   知识管理层（AI 的"大脑"）              │  │
│  │  · 语义等价库（扩充版 gis_ontology）     │  │
│  │  · 标准规则库（标准文件结构化解析）      │  │
│  │  · 数据模型库（EA 仓库对接）            │  │
│  │  · 治理案例库（40个项目经验沉淀）       │  │
│  ├────────────────────────────────────────┤  │
│  │   底座调用层（对接时空数据治理平台）      │  │
│  │  · 数据源管理 API                       │  │
│  │  · 元数据读取 API                       │  │
│  │  · 质检/治理执行 API                    │  │
│  │  · 数据资产/数据服务 API                │  │
│  └────────────────────────────────────────┘  │
├──────────────────────────────────────────────┤
│       时空数据治理平台（已有底座）              │
│  数据源 / 元数据 / 标准 / 模型导入 / 质检 /   │
│  算子 / 编排 / 调度 / 资产 / 服务 / BI        │
└──────────────────────────────────────────────┘
```

### 各层职责边界

| 层 | 职责 | 不做什么 |
|---|------|---------|
| **智能交互层** | 理解用户意图、调用知识库、编排底座能力、生成报告 | 不直接操作数据库、不重复实现底座已有功能 |
| **知识管理层** | 存储和检索领域知识、支撑 AI 推理的准确性 | 不做数据治理的执行逻辑 |
| **底座调用层** | 封装时空数据治理平台的 API 为 GIS Data Agent 可调用的接口 | 不修改底座内部逻辑 |

### 3.4 需要新增的模块

对照 C-分析中 10 个缺失的核心域概念和三层架构，需要新增：

#### 知识管理层

| 新增模块 | 对应核心域概念 | 职责 |
|---------|-------------|------|
| `knowledge/semantic_vocab.py` | 语义等价库（扩充版） | 管理 GIS 语义等价关系，从 37 组扩充到覆盖自然资源全业务；统一 `gis_ontology.yaml` + DB 注册 + 内存缓存为一个模块 |
| `knowledge/standard_rules.py` | 标准规则库 | 从数据标准文件（PDF/Word/YAML）中智能解析出结构化规则条目；对接底座的标准管理模块 |
| `knowledge/model_repo.py` | 数据模型库 | 对接 EA 在线仓库，读取通用数据模型结构（业务域 × 实体对象矩阵）；缓存模型结构供 AI 推理使用 |
| `knowledge/case_library.py` | 治理案例库 | 存储"数据现状→治理方案→执行结果"的历史案例；支持相似案例检索；操作人员的反馈（接受/拒绝/调整 AI 建议）回流为新案例 |

#### 智能交互层

| 新增模块 | 对应用例 | 职责 |
|---------|---------|------|
| `intelligence/profiler.py` | UC-02 数据画像 | 调用底座元数据 API + 自动统计分析，生成数据画像；调用语义等价库识别字段含义 |
| `intelligence/gap_analyzer.py` | UC-04 标准对照 | 将数据画像与标准规则库对照，生成差距报告（含差距项列表） |
| `intelligence/model_advisor.py` | UC-05 模型推荐 | 基于差距报告 + 数据模型库 + 治理案例库，推荐数据模型调整方案 |
| `intelligence/governance_orchestrator.py` | UC-06 治理执行 | 基于确认后的模型和标准，推荐并编排治理流程；调用底座执行 |
| `intelligence/report_generator.py` | UC-07 成果报告 | 汇总治理前后质量指标对比，生成可验收的 Word/PDF 报告 |

#### 底座调用层

| 新增模块 | 职责 |
|---------|------|
| `platform/datasource_api.py` | 封装底座的数据源管理 API |
| `platform/metadata_api.py` | 封装底座的元数据读取 API |
| `platform/governance_api.py` | 封装底座的质检/治理执行 API（汇聚、质检、开发、服务发布、资产登记） |
| `platform/asset_api.py` | 封装底座的数据资产和数据服务 API |

## 5. 实施路径

### Phase 1：P0 验证闭环（面向山西总工演示）

**目标**：在山西项目中选一个具体的数据治理子场景，跑通 UC-05 → UC-06 → UC-07 → UC-10 闭环。

**新增模块**：
- `knowledge/model_repo.py` — 对接 EA 仓库读取通用模型
- `intelligence/model_advisor.py` — 基于模型 + 人工提供的现状描述生成调整建议
- `intelligence/governance_orchestrator.py` — 调用底座执行治理
- `intelligence/report_generator.py` — 生成治理成果报告
- ChatBI — 已在并行开发

**新增 DB 表**：
- `agent_gap_reports` + `agent_gap_items`
- `agent_model_adjustments`
- `agent_governance_reports`

**知识注入**：
- 将山西项目涉及的那个子场景的数据标准手动导入标准规则库
- 将对应的通用数据模型从 EA 仓库导出为 AI 可读格式

**验证标准**：一个完整场景从"推荐模型调整"到"出验收报告"能走通，总工认为有价值。

### Phase 2：P1 操作者提效（解决步骤 1-4 真空）

**目标**：让实施人员能用 AI 完成数据现状分析和标准对照，不再纯靠人脑。

**新增模块**：
- `knowledge/semantic_vocab.py` — 语义等价库从 37 组扩充
- `knowledge/standard_rules.py` — 标准文件智能解析
- `intelligence/profiler.py` — 自动数据画像
- `intelligence/gap_analyzer.py` — 自动标准对照

**新增 DB 表**：
- `agent_data_profiles` + `agent_quality_metrics`

**知识注入**（核心工作量）：
- 从 40 个项目经验中提炼语义等价关系，扩充 `gis_ontology.yaml`
- 把常用的数据标准文件做结构化解析入库

**验证标准**：实施人员不需要人肉逐字段对照，系统自动生成差距报告的准确率达到可用水平。

### Phase 3：P2 持续运营 + 经验飞轮

**目标**：系统越用越准，知识库持续增长。

**新增模块**：
- `knowledge/case_library.py` — 治理案例库
- 数据质量变化监控（UC-08）
- 数据服务发布对接（UC-09）

**知识注入**：
- 操作人员对 AI 建议的反馈（接受/拒绝/调整）自动回流为案例
- 新项目的治理经验持续积累

## 6. CLAUDE.md 更新建议

在现有 CLAUDE.md 中增加以下内容：

```markdown
## 产品定位（A-业务建模）

GIS Data Agent 是时空数据治理平台的智能化入口。目标：让能力不足的
实施人员也能完成数据治理项目交付。

第一验证场景：山西省测绘院全域数据治理项目。

## 架构分层

三层架构：智能交互层 → 知识管理层 → 底座调用层 → 时空数据治理平台。
GIS Data Agent 不重建底座已有能力，通过 API 调用。

## 核心域约束（C-分析层）

21 个核心域概念，详见 docs/C_分析_迭代1.md。

### 新增模块的命名规则
- knowledge/ — 知识管理层模块
- intelligence/ — 智能交互层模块
- platform/ — 底座调用层模块

### 新增 DB 表规则
- 表名前缀 agent_
- 每张新表必须对应一个核心域概念
- 不允许出现表达同一概念的两张表

### 开发约束
- 底座已有的能力（数据源管理、元数据、算子、调度等）通过
  platform/ 层 API 调用，不在 GIS Data Agent 中重新实现
- AI 推理必须基于知识管理层的结构化知识，不允许让 LLM 
  无依据地"猜"领域知识
- 操作人员对 AI 建议有最终决定权，系统不做自主决策
```

## 7. 与之前 47 天开发方式的对比

| 维度 | 之前 | 改进后 |
|------|------|--------|
| **开发驱动力** | 对标竞品 + 功能列表 | 愿景驱动 + 用例分层 |
| **架构组织** | 按技术活动（toolsets/探索/分析/可视化） | 按三层架构（智能交互/知识管理/底座调用） |
| **新增表的依据** | "这个功能需要存数据" | "这个概念在核心域类图上是什么" |
| **AI 的角色** | 直接用 LLM 回答问题 | LLM + 知识库，知识驱动推理 |
| **与底座的关系** | 独立系统，重复实现部分能力 | 底座的智能化入口，调用不重建 |
| **版本节奏** | 每周一个大版本，功能堆积 | 按 Phase 推进，每阶段有明确验证标准 |
| **知识积累** | 代码即知识（写完就散落了） | 结构化知识库，越用越准 |
