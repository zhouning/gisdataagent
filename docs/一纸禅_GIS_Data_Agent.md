# 【一纸禅】GIS Data Agent 地理空间智能分析平台

**版本**：v16.0 &nbsp;|&nbsp; **更新日期**：2026-04-01

---

## 一句话描述

GIS Data Agent 是一个用自然语言驱动的地理空间分析平台。用户用中文或英文描述分析需求，系统自动路由到对应的专业管线执行——空间优化、数据治理、通用分析——直接输出地图、报告和数据表格。不需要写代码，不需要点菜单，说清楚要什么就行。

---

## 适用场景

- **自然资源数据治理**：耕地/林地数据的拓扑检查、字段标准校验、缺陷检测与自动修复
- **国土空间布局优化**：基于深度强化学习的耕地连片整治、设施选址、生态补偿方案生成
- **测绘成果质量检验**：按 GB/T 24356 标准的 30 项缺陷编码检测，支持 DLG/DOM/DEM/三维模型全品类
- **多源数据融合分析**：矢量×矢量、矢量×栅格、栅格×栅格的 10 种融合策略，自动选择最优方案
- **因果关系挖掘**：土地利用变化的驱动因素识别——倾向得分匹配、双重差分、Granger 因果检验等 6 种方法

## 它实际能做什么

### 说人话的 GIS 分析

用户输入自然语言，系统用 Gemini 2.0 Flash 做语义路由，分到三条管线：

| 管线 | 做什么 | 典型输入 |
|------|--------|---------|
| **优化管线** | 数据探索 → 处理 → DRL 优化 → 可视化 → 报告 | "帮我优化这批耕地的布局，减少破碎度" |
| **治理管线** | 拓扑审计 → 字段校验 → 缺陷修复 → 治理报告 | "检查这个图层的拓扑错误和字段规范性" |
| **通用管线** | 空间分析 → 可视化 → 摘要 | "画一张土地利用类型的专题图" |

每条管线内置质量自检循环：Generator 产出结果 → Critic 检查质量 → 不合格则重试（最多 3 轮）。不是"生成就完事"，而是自己检查自己。

### 40 个工具集，240+ 个工具函数

不是概念，是实际的 Python 函数。举几个例子：

| 工具集 | 干什么 | 工具数 |
|--------|--------|--------|
| GeoProcessingToolset | 裁剪、缓冲、叠加、重投影、聚类 | 26 |
| GovernanceToolset | 拓扑检查、字段标准校验、缺陷检测 | 18 |
| DataCleaningToolset | 去重、填空、标准化、异常值处理 | 11 |
| CausalInferenceToolset | PSM、DiD、Granger、Causal Forest | 6 |
| VisualizationToolset | 交互地图、专题图、3D 拉伸、图层控制 | 15 |
| ChartToolset | ECharts 柱状/折线/饼/散点/雷达/桑基图 | 9 |

完整列表见技术文档第 4 章。

### 用户自己扩展，不改代码

三种扩展机制，都通过 Web 界面操作：

1. **自定义技能**：写一段指令文本 + 选几个工具集 = 一个新的 AI Agent（支持版本管理、评分、克隆、审批发布）
2. **声明式工具**：填模板创建工具——调 REST API、跑 SQL 查询、文件处理管道、串联多个工具（每用户最多 50 个）
3. **工作流编排**：拖拽式 DAG 编辑器，把管线和技能串成多步骤流程，支持 Cron 定时、条件分支、SLA 超时告警

### 地图渲染

- **2D**：Leaflet.js，支持专题图、气泡图、分级设色、分类设色，底图支持高德/天地图/CartoDB/OSM
- **3D**：deck.gl + MapLibre GL，支持 3D 拉伸多边形、柱状图、弧线连接图

## 架构选择与取舍

说清楚我们选了什么，也说清楚我们没做什么：

**选了的**：
- **Google ADK** 作为 Agent 框架（不是自研框架，站在巨人肩上）
- **Gemini 2.5 Flash/Pro** 作为推理模型（Flash 做日常分析，Pro 做复杂报告）
- **MCP + A2A** 开放协议（可以被外部 Agent 调用，也可以调用外部工具服务器，最多 20 个）
- **PostgreSQL + PostGIS** 单库架构（够用，不过度工程）
- **Chainlit** 做 WebSocket 通信层（不重复造轮子）

**没做的 / 有局限的**：
- 知识图谱是内存中的 NetworkX，没有持久化到 Neo4j——大数据集（10 万+ 实体）会撑不住
- Cron 调度在内存里，进程重启后需要重新同步
- 前端状态管理还是 props drilling，没上 Zustand——组件层级深的地方维护成本高
- DRL 模型权重需要预训练，不是开箱即用——需要 `scorer_weights_v7.pt` 权重文件
- World Model (AlphaEarth + JEPA) 是技术预览阶段，训练数据有限

## 与超图 AgentX Server 的定位差异

不做直接对比评测（信息不对称），只说定位差异：

| 维度 | GIS Data Agent | SuperMap AgentX Server |
|------|----------------|----------------------|
| **定位** | 面向分析师的端到端分析工具 | 面向开发者的智能体搭建平台 |
| **交互方式** | 自然语言 → 直接出结果 | 低代码拖拽搭建智能体 |
| **GIS 引擎** | 开源栈（GeoPandas/Shapely/Rasterio）+ 可选 ArcPy | SuperMap iServer/iDesktop 全家桶 |
| **模型** | Google Gemini（云端 API） | 支持多种大模型接入 |
| **独有能力** | DRL 优化、三角度因果推断、World Model | 门户GIS/服务器GIS/GPA 全产品线 MCP 工具 |
| **开源性** | 核心代码开源 | 商业产品 |
| **适用规模** | 单机/小团队（PostgreSQL 单库） | 企业级（集群部署） |
