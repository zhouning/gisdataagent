## 演示环节一：时序对齐能力 (3 分钟) ⭐ 核心

### 场景设定

多期土地利用数据融合——同一区域 2023 年和 2024 年的地块数据，时间格式不统一（一个用 ISO8601，一个用中文日期），需要检测哪些地块发生了变化。

### 操作步骤

1. 在聊天面板上传两个文件：`parcels_2023.geojson` 和 `parcels_2024.geojson`
2. 输入：**"帮我检查这两个文件的时间列是否一致"**
3. 系统通过 General Pipeline 调用 `validate_temporal_consistency` 工具
4. 输出时序一致性报告：检测到时间列名、格式差异、是否有空值/重复
5. 输入：**"将两个文件的时间戳标准化为 UTC 格式"**
6. 系统调用 `standardize_timestamps` 工具，输出标准化后的文件路径

### 展示要点

- **自动检测时间列**：系统识别 `调查时间`、`timestamp`、`date` 等中英文时间列名
- **异构格式兼容**：支持 ISO8601、`YYYY/MM/DD`、`YYYY年MM月DD日`、epoch 等 10+ 格式
- **一致性报告**：total_records、parsed_count、time_range、issues 列表
- **标准化输出**：统一为 `_std_timestamp` 列 (datetime64[ns, UTC])

### 话术

> "传统融合引擎对时间维度的处理仅限于简单过滤。v2.0 的时序对齐层可以自动检测时间列（支持中英日多语言列名），将 ISO8601、中文日期、Unix 时间戳等异构格式统一为 UTC。更重要的是，它能生成完整的时序一致性报告——有多少条记录解析成功、是否存在时间间断或重复。这为后续的多期变化检测和时空融合打下基础。"

## 演示环节二：GIS 领域本体推理 (3 分钟)

### 场景设定

两个来源的建筑数据需要融合——A 源有 `面积`、`层数`，B 源有 `AREA`、`building_height`。传统融合引擎无法识别「面积」和「AREA」语义等价，更无法推导出 A 源缺失的建筑高度。

### 操作步骤

1. 上传两个建筑数据文件
2. 输入：**"评估这两个文件的融合兼容性，启用本体匹配"**
3. 系统调用 `assess_fusion_compatibility`（use_ontology 参数启用）
4. 展示兼容性报告：
   - Tier 1 精确匹配：0 对（字段名完全不同）
   - **Tier 1.5 本体匹配：2 对（面积↔AREA, 层数↔building_floors）**，置信度 0.85
5. 输入：**"融合这两个数据集"**
6. 系统执行融合，本体推理自动推导缺失字段

### 展示要点

- **本体等价组**：展示 `standards/gis_ontology.yaml` 中的 15 个等价组
  - `area` 组: ["area", "面积", "mj", "zmj", "shape_area", "TBMJ", "AREA"]
  - `building_floors` 组: ["floors", "层数", "cs", "floor_count"]
- **推导规则**：`building_height = floors × 3.0`（A 源有层数但无高度，自动推导）
- **推理规则**：`floors ≥ 18 → building_type = "高层住宅"`（条件推理）
- **7 层匹配层级**：精确→本体→等价组→LLM Schema→嵌入→单位感知→模糊

### 话术

> "这是 v2.0 的核心创新之一——GIS 领域本体推理。系统内置了一个包含 15 个等价组、8 条推导规则、5 条推理规则的 GIS 本体知识库。它知道'面积'和'AREA'和'mj'都是同一个语义概念。更强大的是推导能力——如果 A 源有'层数'但缺'建筑高度'，系统自动应用公式 `building_height = floors × 3.0` 补全缺失字段。这在传统融合中需要人工编写脚本处理，现在是全自动的。"

## 演示环节三：多源冲突解决 (3 分钟) ⭐ 核心

### 场景设定

三个来源的地块数据描述同一区域——测绘局数据（权威但较旧）、遥感提取数据（最新但精度一般）、规划部门数据（高精度但局部覆盖）。同一地块的面积值在三个来源中不一致。

### 操作步骤

1. 上传或指定三个数据源
2. 输入：**"融合这三个数据集，面积冲突时按数据源优先级解决，测绘局优先级最高"**
3. 系统：
   - 融合后检测到 AREA 列在 `_left` / `_right` 间存在冲突
   - 使用 `source_priority` 策略，测绘局值优先
   - 计算逐要素置信度分数
   - 添加 `_source_AREA` 列标注每个值的来源
4. 输出结果摘要：X 处冲突已解决，策略=source_priority，平均置信度=Y

### 展示要点

- **6 种消解策略**（选其一演示）：
  | 策略 | 适用场景 | 本次演示 |
  |------|---------|---------|
  | source_priority | 权威数据源已知 | ✅ 主演示 |
  | latest_wins | 时效性优先 | 可提及 |
  | voting | 多源等权（数值取均值） | 可提及 |
  | llm_arbitration | 复杂模糊冲突 | 提及但不演示 |
  | spatial_proximity | 空间精度加权 | 提及 |
  | user_defined | 业务定制 | 提及 |

- **置信度评分**：`confidence = 0.4×timeliness + 0.3×precision + 0.3×completeness`
  - 冲突行自动降权（降低 0.3）
  - 无冲突行置信度=1.0

- **来源标注**：`_source_AREA` 列记录每个值最终来自哪个数据源

### 话术

> "多源数据融合最常见的痛点就是值冲突——同一个地块，三个来源给出三个不同的面积值。v2.0 提供 6 种消解策略。今天演示的是 source_priority：预设数据源优先级（测绘局=10 > 遥感=5 > 规划=3），冲突时取优先级最高的值。系统会自动计算每个要素的置信度分数——无冲突的要素置信度为 1.0，有冲突的会根据数据源质量降权。同时，`_source_AREA` 列会标注每个值的实际来源，实现全链路可追溯。"

## 演示环节四：融合可解释性 (3 分钟) ⭐ 核心

### 场景设定

融合完成后，用户需要理解：哪些区域融合质量好、哪些区域有问题、每个要素的数据来自哪里。

### 操作步骤

1. 使用前面融合完成的结果
2. 输入：**"生成这个融合结果的质量热力图"**
3. 系统输出：
   - 质量热力图 GeoJSON（置信度三色分级：红<0.3 / 黄0.3-0.7 / 绿>0.7）
   - 地图面板自动渲染热力图层
4. 在地图上点击某个低置信度（红色）要素
5. 查看该要素的详情：
   - `_fusion_confidence`: 0.25
   - `_fusion_sources`: ["测绘局_2023.shp", "遥感_2024.tif"]
   - `_fusion_method`: "spatial_join"
   - `_fusion_conflicts`: {"AREA": 1}

### 展示要点

- **逐要素元数据**（4 列注入每一行）：
  - `_fusion_confidence`: 0-1 置信度
  - `_fusion_sources`: 来源文件列表 (JSON)
  - `_fusion_method`: 使用的策略名
  - `_fusion_conflicts`: 冲突详情 (JSON)

- **质量热力图**：
  - low (红): confidence < 0.3 — 需要人工审查
  - medium (黄): 0.3 ≤ confidence < 0.7 — 可能需要关注
  - high (绿): confidence ≥ 0.7 — 质量可靠
  - 大数据集 (>100K 要素) 自动简化几何，避免渲染卡顿

- **决策解释**：自然语言描述（模板化，非 LLM）
  > "该要素由 测绘局_2023.shp、遥感_2024.tif 融合生成，使用 spatial_join 策略。置信度: 0.25 (low)。存在 1 个属性冲突，已按规则解决。"

### 话术

> "这是用户感知最强的升级——融合结果不再是一个黑盒。每个要素都携带了 4 个可解释性列：置信度、来源、策略、冲突。质量热力图一目了然地展示哪些区域融合质量好（绿色）、哪些需要人工审查（红色）。点击任意要素，可以看到完整的决策解释——数据来自哪里、用什么策略融合的、是否有冲突。这在向甲方交付成果时极其重要——每一条数据都可追溯、可解释。"

## 演示环节六：REST API 验证 (2 分钟)

### 操作步骤

在浏览器或终端中依次访问 5 个新 API：

```bash
# 1. 列出所有融合操作（含 v2 特性标记）
curl http://localhost:8000/api/fusion/operations

# 2. 查看单次操作的质量详情 + 可解释性
curl http://localhost:8000/api/fusion/quality/1

# 3. 查看融合溯源链
curl http://localhost:8000/api/fusion/lineage/1

# 4. 查看冲突消解日志
curl http://localhost:8000/api/fusion/conflicts/1

# 5. 时序对齐预览（不执行融合，仅分析）
curl -X POST http://localhost:8000/api/fusion/temporal-preview \
  -H "Content-Type: application/json" \
  -d '{"file_path": "uploads/admin/parcels_2023.geojson"}'
```

### 展示要点

- 所有 API 返回结构化 JSON
- JWT cookie 认证（登录后自动携带）
- `temporal-preview` 端点可用于前端集成时序分析预览

### 话术

> "5 个新 REST API 提供了完整的编程接口。`/operations` 列出所有融合操作并标记 v2 特性；`/quality/{id}` 返回可解释性元数据；`/lineage/{id}` 返回完整的融合溯源链——从原始数据源到最终结果的每一步都可追溯；`/conflicts/{id}` 返回冲突消解日志；`/temporal-preview` 可以在不执行融合的情况下预览时序对齐效果，非常适合集成到前端做交互式分析。"

## 总结 (2 分钟)

### v2.0 核心指标

| 指标 | 数值 |
|------|------|
| 新增模块 | 6 个 Python 模块 (~1700 行) |
| 新增测试 | 84 个（214 个 fusion 测试全通过） |
| 新增 API | 5 个 REST 端点 |
| 新增工具 | 2 个 Toolset 工具 |
| 本体知识 | 15 等价组 + 8 推导规则 + 5 推理规则 |
| 冲突策略 | 6 种 |
| 匹配层级 | 7 层（v1.0 为 5 层）|
| 向后兼容 | 100%（所有 v2 功能 opt-in）|

### 能力对比

| 能力 | v1.0 (v7.1) | v2.0 (v17.0) |
|------|-------------|-------------|
| 时序处理 | 仅时间过滤/聚合 | 多时区标准化 + 3 种插值 + 变化检测 |
| 字段匹配 | 名称相似度 (5 层) | +本体推理 +LLM 语义 (7 层) |
| 冲突处理 | 简单覆盖 | 6 策略 + 置信度 + 来源标注 |
| 结果解释 | 全局 10 分制 | 逐要素元数据 + 热力图 + 溯源 |
| 知识利用 | 硬编码等价组 | YAML 本体 + KG + LLM |

### 技术创新点

> "v2.0 有 4 个值得关注的技术创新点：
> 1. **7 层递进式语义匹配**——在精确匹配和传统等价组之间插入本体推理层（Tier 1.5），将匹配准确率从名称相似度提升到语义理解级别；
> 2. **6 策略冲突消解 + 置信度评分**——业界首次在开源 GIS 融合引擎中实现多策略冲突消解与逐要素置信度量化；
> 3. **逐要素可解释性**——每个融合结果要素都携带来源、策略、冲突信息，实现全链路可追溯；
> 4. **GIS 领域本体推理**——基于 YAML 本体的字段推导（如 `building_height = floors × 3.0`）和条件推理（如坡度→地形分类），将领域知识编码为可执行规则。"
