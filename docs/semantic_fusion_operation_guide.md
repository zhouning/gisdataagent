# 多模态时空数据治理与语义融合 — 操作指南

> 基于 GIS Data Agent v18.5+ 的「数据治理」+「多模态语义融合 v2.0」能力，对 `01数据样例` 中的重庆市多源规划数据进行端到端的治理与融合操作。

## 1. 概述

### 1.1 目标

将 12 类异构数据源（矢量、栅格、表格、CAD、文档）经过质量审计、标准化清洗后，通过语义融合引擎合并为统一的城市分析数据集。

### 1.2 系统能力矩阵

| 能力模块 | 工具数 | 核心功能 |
|---------|:------:|---------|
| 数据治理 | 15 | 7 项审计 + 6 维评分 + 缺陷分类 |
| 数据清洗 | 8 | 空值填充、字段映射、CRS 标准化、自动修复 |
| 语义融合 v2.0 | 7 | 10 策略 + 4 层语义匹配 + 时序对齐 + 冲突解决 + 可解释性 |
| 格式转换 | 1 | GDB/Shapefile/GeoJSON/GPKG 互转 |
| 文档上下文 | 1 | PDF/Word/Excel 元数据提取 → 融合上下文 |
| CAD 解析 | 3 | DXF 解析 + DXF→GeoJSON 转换 + 3D 拓扑验证 |

### 1.3 数据覆盖评估

| 数据类别 | 格式 | 大小 | 融合兼容性 | 处理方式 |
|---------|------|------|:---------:|---------|
| DEM 高程 | GeoTIFF | 2.3M | ✅ 直接 | zonal_statistics / point_sampling |
| OSM 道路 | Shapefile | 25M | ✅ 直接 | spatial_join / nearest_join |
| 遥感解译 | Shapefile+TIF | 9.5M | ✅ 直接 | spatial_join / zonal_statistics |
| 建筑轮廓 | Shapefile | 22M | ✅ 直接 | 核心融合基底 |
| 历史文化街区 | Shapefile | 377K | ✅ 直接 | overlay / spatial_join |
| 高德 POI 2024 | GDB | 223M | ⚠️ 需转换 | convert_format → GeoJSON |
| 百度 AOI | GDB | 17M | ⚠️ 需转换 | convert_format → GeoJSON |
| 手机信令 | CSV | 72K | ✅ 直接 | time_snapshot / attribute_join |
| 人口数据 | Excel | 16K | ✅ 直接 | attribute_join |
| 百度搜索指数 | GDB | 323K | ⚠️ 需转换 | convert_format → GeoJSON |

| CAD 设计图 | DWG | ~50M | ⚠️ 需转换 | 先另存 DXF → CAD 解析器转 GeoJSON |
| 规划文档 | PDF/Word | ~20M | 📄 上下文 | inject_document_context 提取元数据 |

### 2.2 GDB 格式预转换

系统内置了 `convert_format` 工具，可直接在聊天中完成 GDB 转换。

**方式一：系统内置转换（推荐）**

1. 将 GDB 目录压缩为 ZIP 文件（如 `高德POI.zip`）
2. 在聊天界面上传 ZIP 文件
3. 输入指令：

```
请将这个 GDB 数据转换为 GeoJSON 格式，导出所有图层
```

系统会自动调用 `convert_format` 工具，逐层导出为独立的 GeoJSON 文件。

> **技术注解**：`convert_format()` 使用 fiona + GeoPandas 读取 GDB（OpenFileGDB 驱动），支持单层/多层导出。输出到用户沙箱目录 `uploads/{user_id}/`。

**方式二：外部工具转换**

如果 GDB 文件过大（>100MB 上传限制），可使用 QGIS 或 ogr2ogr 预转换：

```bash
# ogr2ogr 命令行（需安装 GDAL）
ogr2ogr -f "GeoJSON" output.geojson input.gdb layer_name

# 批量导出所有图层
for layer in $(ogrinfo -so input.gdb | grep "Layer" | awk '{print $2}'); do
    ogr2ogr -f "GeoJSON" "${layer}.geojson" input.gdb "$layer"
done
```

QGIS 操作：打开 GDB → 右键图层 → 导出 → 另存为 GeoJSON。

### 2.3 DWG 预转换

DWG 格式需先转为 DXF：
1. 在 AutoCAD 中打开 DWG 文件
2. 另存为 → 选择 DXF 格式（R2013 或更高版本）
3. 上传 DXF 文件到系统

上传后输入：
```
请将这个 DXF 文件转换为 GeoJSON，坐标系设为 EPSG:4490
```

系统通过 CAD 解析器（MCP 服务）完成 DXF → GeoJSON 转换。

> **技术注解**：CAD 解析器使用 ezdxf 库解析 DXF 实体（LINE, POLYLINE, CIRCLE, POINT, TEXT），通过 shapely 构建几何对象，支持指定目标 CRS。

## 3. Phase 1：单数据源治理

### 3.1 操作流程

```
上传文件 → 自动预览 → 输入治理指令 → 确认分析方案 → 查看审计报告
```

### 3.2 步骤详解

**Step 1：上传数据文件**

- 点击聊天输入框左侧的 📎 按钮，或直接拖拽文件到聊天区域
- 支持格式：`.shp`（需连同 .dbf/.prj/.shx 一起打包为 ZIP）、`.geojson`、`.gpkg`、`.csv`、`.xlsx`
- 单文件限制：100MB
- ZIP 文件会自动解压并识别空间文件

**Step 2：查看自动预览**

上传成功后，系统自动显示数据概览：
- 文件名、大小、格式
- 坐标系（CRS）、要素数量、几何类型
- 字段列表和类型
- 前 5 行数据样本

**Step 3：输入治理指令**

在聊天框中输入以下任一指令（关键词触发 GOVERNANCE 管道）：

```
对这份数据进行数据治理
```

```
请对上传的建筑轮廓数据执行全面质量审计，检查拓扑、完整性和坐标系合规性
```

```
按照 GB/T 24356 标准对这份测绘数据进行质检
```

> **触发关键词**：治理、审计、质检、核查、拓扑、标准 → 系统自动路由到 GOVERNANCE 管道

**Step 4：确认分析方案**

系统会生成分析方案并展示，包含：
- 计划执行的审计项目
- 预计使用的工具
- 参考标准

点击「确认」执行，或点击「修改」调整方案。

**Step 5：查看审计报告**

治理管道自动执行 4 个阶段：

1. **探索审计**：7 项检查（拓扑、间隙、完整性、属性范围、重复、CRS、字段标准）
2. **数据修复**：CRS 统一、几何修复、字段标准化、去重
3. **可视化**：雷达图（6 维评分）、问题分布地图、完整性柱状图
4. **报告生成**：Markdown 格式审计报告（含评分、缺陷列表、修复建议）

### 3.3 治理 6 维评分说明

| 维度 | 权重 | 检查内容 |
|------|:----:|---------|
| 拓扑 | 25% | 重叠、自相交检测 |
| 间隙 | 15% | 多边形间隙数量和面积 |
| 完整性 | 20% | 字段和几何完整率 |
| 属性有效性 | 15% | 值域合规率 |
| 重复 | 10% | 几何和属性重复 |
| 坐标系 | 15% | CRS 合规性（目标：CGCS2000/EPSG:4490） |

评级：A(≥90) B(≥80) C(≥60) D(≥40) F(<40)

### 3.4 推荐治理顺序

建议按以下顺序逐个治理核心数据源：

1. **建筑轮廓** — 作为融合基底，优先确保质量
2. **OSM 道路** — 检查拓扑连通性
3. **遥感解译** — 验证分类编码合规性
4. **历史文化街区** — 检查边界完整性
5. **DEM 高程** — 验证空间范围和分辨率

## 4. Phase 2：两源语义融合

### 4.1 融合操作通用流程

```
上传 2 个文件 → 输入融合指令 → 系统自动：画像→兼容性评估→策略选择→执行→质量验证
```

### 4.2 七个典型融合场景

#### 场景 1：DEM + 建筑轮廓 → 高程赋值（zonal_statistics）

**目标**：为每栋建筑计算平均高程、最高点和最低点。

**上传文件**：
- `01重庆市DEM数据/chongqing_dem.tif`
- `04重庆市中心城区建筑数据/buildings.shp`（ZIP 打包）

**聊天指令**：
```
请融合 DEM 高程数据和建筑轮廓数据，计算每栋建筑的平均高程、最高点和最低点
```

**预期结果**：
- 策略：`zonal_statistics`（自动选择，因为是矢量×栅格）
- 输出：GeoJSON，每栋建筑新增 `raster_mean`、`raster_min`、`raster_max` 字段
- v2 特性：`_fusion_confidence` 置信度列 + 质量热力图

---

#### 场景 2：OSM 道路 + 建筑轮廓 → 距离计算（nearest_join）

**目标**：计算每栋建筑到最近道路的距离。

**上传文件**：
- `02重庆市OSM道路数据/OSM_roads.shp`（ZIP 打包）
- `04重庆市中心城区建筑数据/buildings.shp`（ZIP 打包）

**聊天指令**：
```
融合道路数据和建筑数据，计算每栋建筑到最近道路的距离
```

**预期结果**：
- 策略：`nearest_join`（自动选择，因为道路是线、建筑是面，空间重叠低）
- 输出：每栋建筑新增 `distance` 字段（单位：度，需后续转换为米）

---

#### 场景 3：遥感解译 + 建筑轮廓 → 土地利用关联（spatial_join）

**目标**：为每栋建筑标注所在地块的土地利用类型。

**上传文件**：
- `03重庆市遥感影像解译数据/land_use.shp`（ZIP 打包）
- `04重庆市中心城区建筑数据/buildings.shp`（ZIP 打包）

**聊天指令**：
```
将遥感解译的土地利用数据与建筑轮廓进行空间连接，为每栋建筑标注土地利用类型
```

**预期结果**：
- 策略：`spatial_join`（自动选择，两个矢量面数据有空间重叠）
- 输出：建筑数据新增土地利用分类字段
- 语义匹配：4 层渐进式匹配自动对齐中文字段名

---

#### 场景 4：历史文化街区 + 建筑轮廓 → 保护区识别（overlay）

**目标**：识别位于历史文化保护区内的建筑。

**上传文件**：
- `05历史文化街区数据/historic_districts.shp`（ZIP 打包）
- `04重庆市中心城区建筑数据/buildings.shp`（ZIP 打包）

**聊天指令**：
```
将历史文化街区范围与建筑轮廓进行叠加分析，识别保护区内的建筑
```

**预期结果**：
- 策略：`overlay`（intersection 操作）
- 输出：仅保留与保护区相交的建筑，保留双方属性

---

#### 场景 5：建筑轮廓 + 人口数据 → 人口密度关联（attribute_join）

**目标**：将各区县人口数据关联到建筑数据。

**上传文件**：
- `04重庆市中心城区建筑数据/buildings.shp`（ZIP 打包）
- `08各区县人口规模表格数据/population.xlsx`

**聊天指令**：
```
将人口统计表格与建筑数据按区县名称进行属性连接
```

**预期结果**：
- 策略：`attribute_join`（自动选择，因为 Excel 是表格数据）
- 语义匹配：自动识别 "区县" / "district" 等等价字段
- 输出：建筑数据新增人口相关字段

---

#### 场景 6：OSM 道路 + DEM → 道路高程分析（point_sampling）

**目标**：沿道路采样 DEM 高程值，分析道路坡度。

**上传文件**：
- `02重庆市OSM道路数据/OSM_roads.shp`（ZIP 打包）
- `01重庆市DEM数据/chongqing_dem.tif`

**聊天指令**：
```
融合道路矢量数据和 DEM 栅格数据，沿道路采样高程值用于坡度分析
```

**预期结果**：
- 策略：`point_sampling`（自动选择，线矢量×栅格）
- 输出：道路要素新增 `raster_value` 高程采样字段

---

#### 场景 7：手机信令 + 建筑/行政区 → 通勤分析（time_snapshot + attribute_join）

**目标**：将手机信令通勤数据与空间数据关联。

**上传文件**：
- `11中国联通手机信令数据/commute_data.csv`
- `04重庆市中心城区建筑数据/buildings.shp`（ZIP 打包）

**聊天指令**：
```
融合手机信令通勤数据和建筑空间数据，启用时序对齐，分析通勤模式的空间分布
```

**预期结果**：
- 策略：`time_snapshot`（自动检测到 CSV 中的时间列）
- v2 时序对齐：自动检测时间列 → 标准化为 UTC → 时间窗口匹配
- 输出：空间化的通勤数据，含 `_fusion_confidence` 置信度

> **v2 特性说明**：当输入 `enable_temporal="auto"` 时，系统自动扫描所有数据源的列名，匹配 23 种时间列模式（含中文：观测时间、采集时间、调查时间等），检测到后自动启用时序对齐。

## 5. Phase 3：文档辅助融合

### 5.1 目标

利用规划文档（PDF/Word）中的元数据信息，为融合过程提供上下文——包括数据来源说明、字段含义、时间范围、质量声明等，用于冲突解决时的源优先级判定。

### 5.2 操作步骤

**Step 1：上传规划文档**

上传 `07规划编制相关数据` 中的 PDF 或 Word 文件。

**Step 2：提取文档上下文**

```
请从这些规划文档中提取数据元信息，包括数据来源、时间范围、坐标系和质量说明
```

系统调用 `inject_document_context` 工具：
1. PDF → pypdf 文本提取 + Gemini 原生 PDF 理解
2. Word → python-docx 段落提取
3. Excel → pandas 读取表头和摘要
4. Gemini Flash 从文档内容中提取结构化元数据

**Step 3：将元数据用于融合**

提取结果包含每个文档的 `timeliness`（时效性）、`precision`（精度）、`completeness`（完整性）评分，可在后续融合中作为冲突解决权重：

```
融合建筑数据和规划用地数据，使用 source_priority 冲突解决策略，
优先采用规划文档中标注的最新数据源
```

> **技术注解**：`inject_document_context()` 返回的 `source_metadata` 结构可直接对接 `ConflictResolver` 的 `source_metadata` 参数，实现文档驱动的冲突解决。

## 6. Phase 4：多源级联融合

### 6.1 策略

将 Phase 2 的两源融合结果逐步合并为综合数据集：

```
融合结果 1 (建筑+高程) ──┐
                         ├── 级联融合 → 综合城市分析数据集
融合结果 2 (建筑+道路距离) ┘
```

### 6.2 操作步骤

1. 完成 Phase 2 的各场景融合，记录每个输出文件路径
2. 在聊天中输入：

```
请将之前融合生成的建筑高程数据和建筑道路距离数据进行二次融合，
合并为一个综合分析数据集
```

系统会自动识别两个 GeoJSON 文件的共同几何字段，通过 `spatial_join` 合并所有属性。

> **注意**：级联融合时建议启用冲突解决，避免同名字段冲突：
> ```
> 融合这两个数据集，使用 voting 冲突解决策略
> ```

## 8. 常见问题与排障

### Q1：GDB 文件上传失败

GDB 是目录格式，需先压缩为 ZIP 再上传。如果超过 100MB 限制，使用外部工具（QGIS/ogr2ogr）预转换为 GeoJSON。

### Q2：CRS 不一致导致融合失败

错误信息：`CRS mismatch`

解决：先对数据执行治理（自动统一到 CGCS2000），或手动指令：
```
请将这份数据的坐标系转换为 EPSG:4490
```

### Q3：字段匹配失败

如果自动语义匹配未能正确对齐字段，可手动指定：
```
融合这两个数据，使用属性连接，连接字段为 XZQDM
```

### Q4：融合结果为空

可能原因：两个数据源的空间范围不重叠。先检查：
```
分析这两个数据源的空间范围是否有重叠
```

### Q5：DWG 文件无法直接上传

DWG 格式需先在 AutoCAD 中另存为 DXF 格式，然后上传 DXF 文件。

### Q6：大文件处理缓慢

系统对大数据集（>100K 行）自动启用分块处理。如仍然缓慢，建议：
- 先裁剪到研究区域范围
- 使用 PostGIS 推送（系统自动判断）

| `conflict_strategy` | 空（不启用） | 冲突解决：source_priority / latest_wins / voting / llm_arbitration |
| `enable_explainability` | `true` | 可解释性注解：添加置信度、来源、冲突字段 + 质量热力图 |
| `use_llm_semantic` | `false` | LLM 增强语义匹配：Gemini 深度理解字段语义 |
