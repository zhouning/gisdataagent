# DRL 布局优化前端操作演示脚本 — 数据面板「优化」Tab

> 面向导师/评审的前端 UI 操作演示
>
> 演示时长：约 15 分钟
>
> 版本：v17.1 (2026-04-04)
>
> 前提：Data Agent 已启动 (`chainlit run data_agent/app.py -w`)，PostgreSQL 可用，已有含 DLMC + Slope 字段的测试数据

---

## 开场 (1 分钟)

### 背景说明

> "除了通过聊天面板自然语言触发的全自动优化管道，我们还在数据面板中提供了一个独立的「优化」Tab——用户可以在这里手动选择场景、精细调节权重参数、一键运行 DRL 优化，并查看历史记录和对比不同方案的指标差异。这是一个面向高级用户和研究人员的参数调优工作台。"

---

## 演示环节一：进入优化 Tab (1 分钟)

### 操作步骤

1. 打开浏览器访问 `http://localhost:8000`
2. 使用 `admin` / `admin123` 登录
3. 观察右侧数据面板，底部有 **4 个分组按钮**：**数据 | 智能 | 运维 | 编排**
4. 点击 **「编排」** 分组按钮
5. 在该分组的子 Tab 栏中，点击 **「🎯 优化」** Tab
6. 进入后看到 OptimizationTab 的初始空状态页面

### 展示要点

- **分组导航**：优化 Tab 位于「编排」分组下，与「工作流」「模板」Tab 并列
- **空状态提示**：页面中央显示 ⚙ 图标和提示文字："选择场景、输入数据路径并调整权重后运行优化"，以及说明文字"DRL 模型将通过强化学习寻找最优的用地空间布局调整方案"

### 话术

> "数据面板的 Tab 按功能域分为 4 组。优化 Tab 在「编排」分组中，因为 DRL 优化本质上是一种编排能力——编排地块之间的空间调整。进入后可以看到空状态引导，提示用户需要做三件事：选场景、填路径、调权重。"

---

## 演示环节二：选择优化场景 (2 分钟)

### 操作步骤

1. 在页面顶部找到 **「优化场景」** 下拉选择框
2. 点击下拉框，看到系统从 `/api/drl/scenarios` 接口加载的场景列表：
   - **耕地布局优化 (farmland_optimization)** — 默认选中
   - **城市绿地布局 (urban_green_space)**
   - **设施选址优化 (facility_siting)**
3. 选择 **「耕地布局优化 (farmland_optimization)」**（保持默认）
4. 观察下拉框下方的场景描述文字自动更新为：  
   *"优化耕地与林地的空间分布，最小化耕地坡度、最大化连片度"*
5. 切换到 **「城市绿地布局」** 观察描述变化和权重滑块自动变化，然后**切回「耕地布局优化」**

### 展示要点

- **场景列表**：从后端 API `GET /api/drl/scenarios` 动态加载，而非前端硬编码
- **场景描述**：选中场景后，下方灰色小字显示该场景的用途说明
- **权重联动**：切换场景时，下方的 4 个权重滑块会自动重置为该场景的默认值

### 话术

> "场景列表是从后端 API 动态加载的，对应 `drl_engine.py` 中定义的 `DRLScenario` 模板。每个场景预设了不同的源/目标地类映射和权重配置。注意切换场景时，下面的权重滑块会自动跟着变——比如切到城市绿地场景，连片权重会从 500 跳到 1000，因为绿地的连通性比坡度更重要。"

---

## 演示环节三：输入数据路径 (1 分钟)

### 操作步骤

1. 在 **「数据文件路径」** 输入框中，输入测试数据的路径
2. 路径格式示例：
   - 如果是通过聊天上传的文件：`uploads/admin/your_data.shp`
   - 如果是数据库导出的文件：`uploads/admin/query_result_xxxxxxxx.shp`
   - 也支持绝对路径：`D:/adk/data_agent/uploads/admin/test.shp`
3. 输入框中有占位文字提示：*"输入数据文件路径，如 uploads/user/data.shp"*

### 展示要点

- **路径解析**：后端会通过 `_resolve_path()` 自动解析路径，优先查找用户沙箱目录
- **数据要求**：文件必须包含 `DLMC`（地类名称）和 `Slope`（坡度）字段
- **格式支持**：Shapefile (.shp) 和 GeoJSON (.geojson) 均可

### 话术

> "这里填入包含地块矢量数据的文件路径。如果你之前通过聊天面板上传过 Shapefile 或从数据库导出过数据，文件会保存在 `uploads/用户名/` 目录下。你也可以在左侧数据面板的'文件'Tab 里查看已有文件的名称，然后复制过来。关键是数据里要有 DLMC 和 Slope 两个字段。"

## 演示环节四：权重预设与滑块调节 (3 分钟) ⭐ 核心

### 操作步骤

#### 4.1 权重预设按钮

1. 在数据路径输入框下方，找到 **3 个权重预设按钮**（横排排列）：
   - **「平衡模式」** — slope=1000, contiguity=1000, balance=1000
   - **「坡度优先」** — slope=2000, contiguity=500, balance=500
   - **「连片优先」** — slope=500, contiguity=2000, balance=500
2. 依次点击每个预设按钮，观察下方滑块值的变化
3. 最后点击 **「坡度优先」** 预设

#### 4.2 权重滑块精细调节

1. 在灰色背景的权重面板区域中，有 **4 个滑块**：

   | 滑块 | 参数名 | 范围 | 步长 | 当前场景默认值 |
   |------|--------|------|------|----------------|
   | 坡度权重 | slope_weight | 100-3000 | 50 | 1000 |
   | 连片权重 | contiguity_weight | 100-2000 | 50 | 500 |
   | 平衡权重 | balance_weight | 100-2000 | 50 | 500 |
   | 配对奖励 | pair_bonus | 0.1-10.0 | 0.1 | 1.0 |

2. 每个滑块右侧显示 **当前值**（加粗）和 **默认值**（灰色小字标签 `默认: xxx`）
3. 每个滑块标签旁有 **ⓘ 提示图标**，鼠标悬停显示该参数的含义说明：
   - 坡度权重：*"控制地形坡度对优化的影响程度，值越大越倾向于选择平坦区域"*
   - 连片权重：*"控制地块连片性的重要程度，值越大越倾向于形成连续区域"*
   - 平衡权重：*"控制转换前后面积平衡的约束强度，值越大越严格保持面积守恒"*
   - 配对奖励：*"配对转换的额外奖励系数，影响交换操作的优先级"*
4. 拖动 **坡度权重** 滑块到 `2000`
5. 观察：如果权重总和超过 5000 或低于 1500，会出现 **黄色警告条**：
   > ⚠️ 权重总和 xxxx，建议保持在 1500-5000 范围内以获得最佳效果

#### 4.3 重置为默认

1. 点击 **「重置为默认」** 按钮
2. 所有滑块恢复为当前选中场景（耕地布局优化）的默认权重值：1000/500/500/1.0

### 展示要点

- **三层控制**：预设按钮（快速切换）→ 滑块（精细调节）→ 重置按钮（恢复默认）
- **实时反馈**：滑块拖动时数值即时更新，超限时黄色警告即时出现
- **参数校验**：后端 API 会校验权重范围（slope: 100-3000, contiguity: 100-2000, balance: 100-2000, pair_bonus: 0.1-10.0），超出范围返回错误

### 话术

> "这是整个优化 Tab 的核心区域——权重调节面板。我们提供了三层控制：首先是 3 个预设按钮，点一下就能切到'平衡模式'、'坡度优先'或'连片优先'，适合快速对比不同策略；然后是 4 个精细滑块，可以任意调节每个权重；最后有重置按钮，随时回到默认值。
>
> 注意每个滑块旁边有个小 ⓘ 图标，鼠标悬停会显示该参数的物理含义。右侧灰色小字标签显示当前场景的默认值，方便对比你调了多少。如果权重总和过大或过小，系统会给出黄色警告——这不会阻止运行，但提示你当前配置可能不在最佳范围。"

## 演示环节五：运行优化并查看结果 (3 分钟) ⭐ 核心

### 操作步骤

#### 5.1 启动优化

1. 确认已填写数据路径、已选择场景、权重参数已设置好
2. 点击蓝色的 **「运行优化」** 按钮
3. 按钮状态变为灰色禁用态，文字变为 **「优化计算中...」**
4. 等待 DRL 推理完成（取决于数据规模，通常几秒到几十秒）

#### 5.2 查看结果

优化完成后，「运行优化」按钮下方出现**结果面板**（灰色背景区域），显示：

1. **文本摘要**：
   ```
   Optimization Complete (v7).
   Conversions: 86
   Pairs: 43
   Net Change: 0
   Result SHP: uploads/admin/optimized_data_xxxxxxxx.shp
   Visualization: uploads/admin/optimized_map_xxxxxxxx.png
   ```
   - Conversions：总转换次数
   - Pairs：成对置换次数（理想情况 = Conversions / 2）
   - Net Change：耕地净变化数量（0 表示完美平衡）

2. **输出文件路径**：以 `code` 样式高亮显示的文件路径

3. **结果图片**：如果输出路径为 `.png`，直接在面板中内嵌显示优化结果对比图
   - 黄色区域 = 耕地
   - 绿色区域 = 林地
   - 可以直观看到优化前后的空间布局变化

#### 5.3 错误处理

如果输入路径无效或数据缺少必要字段，结果区域会显示**红色错误提示**：
- `"请输入数据文件路径"` — 未填路径
- `"slope_weight 须在 [100, 3000] 范围内"` — 权重超限
- `"Column 'DLMC' not found"` — 数据缺少必要字段

### 展示要点

- **异步执行**：前端通过 `POST /api/drl/run-custom` 提交，后端使用 `asyncio.to_thread` 在线程池中执行 DRL 推理，不阻塞 UI
- **结果展示**：文本摘要 + 文件路径 + 内嵌图片，三位一体
- **关键指标**：
  - `Net Change = 0`：说明模型完美执行了成对置换，耕地总量未变
  - `Pairs ≈ Conversions / 2`：说明几乎每次转换都有配对

### 话术

> "点击'运行优化'后，请求会发到后端 `POST /api/drl/run-custom` 接口。后端在线程池中异步执行 DRL 推理——加载预训练的 MaskablePPO 模型，初始化 Gymnasium 环境，执行 200 步推理。完成后，结果直接在面板中展示。
>
> 重点看几个指标：Conversions 是总转换次数，Pairs 是成对置换次数——如果 Pairs 约等于 Conversions 的一半，说明模型学会了'一出一入'的配对策略。Net Change 是耕地净变化——0 表示完美的面积守恒。下面这张图是优化结果的静态渲染——黄色是耕地、绿色是林地，可以直观看到高坡度区的耕地被转为了林地。"

## 演示环节六：特征重要性分析 (2 分钟)

### 操作步骤

1. 在结果面板下方，找到分割线下的 **「特征重要性分析」** 按钮
2. 点击按钮，按钮文字变为 **「分析中...」**
3. 分析完成后，下方出现两种可能的结果：

**情况 A — 场景特征分析**（未加载完整模型时）：
- 标题：**「场景特征分析」**
- 显示场景描述和关键特征列表（文字说明）

**情况 B — 特征重要性排名**（有模型权重时）：
- 标题：**「特征重要性排名」**
- 以**水平条形图**形式展示 Top 6 特征的重要性：
  - 每行一个特征：左侧特征名，中间进度条，右侧百分比数值
  - 前 3 名使用主题色（蓝色）进度条，后 3 名使用灰色
  - 特征名示例：
    | 特征名 | 含义 |
    |--------|------|
    | slope (坡度) | 地块坡度值 |
    | is_farmland (耕地) | 当前是否为耕地 |
    | neighbor_ratio (邻域比) | 邻域中耕地地块占比 |
    | neighbor_slope (邻域坡度) | 邻域地块平均坡度 |
    | area (面积) | 地块面积 |
    | slope_vs_mean (坡度偏差) | 地块坡度相对全局均值的偏差 |
- 底部可能有摘要文字说明

### 展示要点

- **可解释性**：通过排列重要性（permutation importance）方法分析模型决策依据
- **6 维逐地块特征 + 8 维全局特征**：对应 DRL 环境的观测空间设计
- **API**：`POST /api/drl/explain`，传入当前选中的 `scenario_id`

### 话术

> "特征重要性分析回答了一个关键问题——DRL 模型在做决策时，最看重哪些因素？从结果可以看到，坡度（slope）和邻域耕地比（neighbor_ratio）通常排在前列，这符合我们的预期：模型优先选择高坡度的耕地进行转换，并倾向于在耕地聚集区域维持连片性。这种可解释性对于向规划部门解释 AI 决策非常重要。"

---

## 演示环节七：运行历史与方案对比 (2 分钟) ⭐ 高级

### 操作步骤

#### 7.1 查看运行历史

1. 在特征分析区域下方，找到分割线下的 **「运行历史」** 按钮
2. 点击按钮，按钮文字变为 **「收起历史」**，下方展开历史记录列表
3. 每条记录显示：
   - 两个**单选按钮**（Radio A 和 Radio B，用于选择对比对象）
   - 记录 ID（如 `#1`、`#2`）
   - 场景 ID（如 `farmland_optimization`）
   - 摘要文字前 40 字（如 `Optimization Complete (v7). Conversions: 86...`）
   - 运行时间（如 `2026-04-04T14:30`）
4. 如果暂无历史记录，显示灰色提示：*"暂无历史记录"*

#### 7.2 方案对比

1. 先运行至少 **两次不同参数** 的优化（例如一次"坡度优先"、一次"连片优先"）
2. 在历史列表中，用**左侧第一列**单选按钮选择 Run A（如 `#1`）
3. 用**左侧第二列**单选按钮选择 Run B（如 `#2`）
4. 选择两个不同的运行后，列表下方出现蓝色 **「对比 #1 vs #2」** 按钮
5. 点击对比按钮，下方出现 **指标对比表格**：

   | 列名 | 说明 |
   |------|------|
   | 指标 | 指标名称（如 avg_slope、contiguity、conversions、pairs 等） |
   | Run A | Run A 的指标值 |
   | Run B | Run B 的指标值 |
   | 差异 | Run B - Run A 的差值（正值绿色、负值红色） |

6. 表格中的差异列用颜色编码：
   - **绿色** (+0.xxx)：Run B 指标更好
   - **红色** (-0.xxx)：Run B 指标更差
   - 无颜色 (—)：无法比较

### 展示要点

- **历史持久化**：运行记录存储在 PostgreSQL 的 `drl_run_history` 表中，按用户隔离
- **API 支撑**：
  - `GET /api/drl/history`：获取当前用户的运行历史
  - `GET /api/drl/compare?a=ID&b=ID`：对比两次运行的指标差异
- **决策支撑**：通过对比不同权重配置的指标，帮助用户找到最适合的参数组合

### 话术

> "运行历史和方案对比是这个 Tab 的高级功能。每次运行的参数和结果都会保存到数据库——包括场景 ID、权重配置、输出指标。这样用户可以做系统性的参数调优实验：比如第一次用'坡度优先'跑一遍，第二次用'连片优先'跑一遍，然后通过对比表格直接看两种策略在各指标上的差异。
>
> 对比表格的差异列用红绿色编码——绿色表示 Run B 更好，红色表示更差。这就像一个 A/B 测试控制台，帮助研究人员做定量的敏感性分析。"

## 总结 (1 分钟)

### 优化 Tab 功能矩阵

| 功能 | UI 组件 | 后端 API | 说明 |
|------|---------|----------|------|
| 场景选择 | 下拉选择框 + 描述文字 | `GET /api/drl/scenarios` | 3 种内置场景，动态加载 |
| 数据路径 | 文本输入框 | — | 支持相对/绝对路径 |
| 权重预设 | 3 个按钮 | — | 平衡/坡度优先/连片优先 |
| 权重调节 | 4 个滑块 + 提示 + 警告 | — | 实时联动，范围校验 |
| 运行优化 | 按钮 + 加载态 | `POST /api/drl/run-custom` | 异步执行，结果内嵌展示 |
| 结果展示 | 文本 + 路径 + 图片 | — | 摘要 + SHP + PNG 三件套 |
| 特征分析 | 按钮 + 条形图 | `POST /api/drl/explain` | 排列重要性分析 |
| 运行历史 | 列表 + 双选按钮 | `GET /api/drl/history` | 按用户隔离，DB 持久化 |
| 方案对比 | 表格 + 红绿编码 | `GET /api/drl/compare` | 指标差异量化展示 |

### 操作流程图

```
选择场景 → 输入数据路径 → 调节权重（或选预设）→ 运行优化
                                                    ↓
                                               查看结果
                                              ↙        ↘
                                   特征重要性分析    调参重跑
                                                      ↓
                                               运行历史对比
```

### 话术

> "总结一下：优化 Tab 是一个面向高级用户的 DRL 参数调优工作台。它的核心价值是**让用户无需写代码就能进行系统性的布局优化实验**——选场景、拖滑块、看结果、对比方案，整个流程都在一个面板内完成。配合聊天面板的自然语言全自动管道，形成'自动化 + 可控性'的双通道设计。"
