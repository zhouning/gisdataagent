## 摘要（中文）

自然语言到 SQL 的查询生成技术近年来随着大语言模型的发展取得了显著进展，但现有研究与 benchmark 主要面向传统关系数据库，对可执行空间 SQL 的覆盖仍然有限。与普通数据仓库相比，GIS 数据库在几何类型、空间谓词、坐标参考系以及面积、距离、缓冲、叠加等空间算子方面具有更强的领域特性，使得 schema grounding 与 SQL 生成面临额外挑战。与此同时，面向真实业务的数据平台往往需要同一套自然语言查询框架同时支持空间与非空间场景。针对这一问题，本文提出一种跨域的 NL2Semantic2SQL 框架，通过意图感知路由、语义层解析、schema grounding、few-shot 检索、SQL 后处理与执行期自纠错等机制，实现对 GIS 数据库与通用数据仓库的统一支持。本文在扩展后的 100 题 GIS benchmark 和约 495 题 BIRD 仓库 benchmark 上评估该框架。在 GIS 侧，full pipeline 达到 EX=0.700，显著优于 baseline 的 0.500（McNemar p=0.0002，统计显著），在 Medium（+0.305）和 Robustness（+0.467）类别上增益最大。在 BIRD 侧，full pipeline 达到 EX=0.501，优于 baseline 的 0.474（+0.027），超过 DIN-SQL 外部基线（0.482），但差异未达统计显著性（McNemar p=0.136）。P2 单轮模式将 BIRD token 成本从 32× 降至 7.9×，GIS 侧为 13.6×，是实际部署中需权衡的因素。上述结果表明，带意图感知路由的统一 semantic-to-SQL 框架在 GIS 侧具有统计显著的增益，在仓库侧具有方向性增益，且在两个评测轨道上均优于或持平于 DIN-SQL 外部基线。

## 贡献

1. 本文提出了一种面向 GIS 与非 GIS 双场景的统一 NL2Semantic2SQL 框架，使自然语言查询在空间数据库与普通数据仓库之间共享同一套 semantic grounding—SQL generation—execution correction 主链路，并通过意图感知路由实现算子级规则的条件化注入。框架的核心设计原则是双语跨域支持：意图分类器双语运行，同时识别中文 GIS 关键词和英文仓库表达式（如"how many"、"what is the average"），无需语言特定的模型切换。

2. 本文构建了一套 100 题 GIS-oriented text-to-SQL benchmark（含 Easy/Medium/Hard/Robustness 四个难度层次），并将普通空间 SQL 任务（85 题，空间执行准确率 Spatial EX）与安全/鲁棒性任务（15 题，Robustness Success Rate）分开评价，因为这两类任务衡量的是本质不同的能力；该 benchmark 与约 495 题 BIRD 子集共同构成可执行的双轨评估协议。

3. 本文通过跨域评测分析表明，full pipeline 在 GIS Spatial EX 上达到 0.682 vs. baseline 0.529（McNemar p=0.0072，统计显著，+0.153），在 GIS Robustness Success Rate 上达到 0.800 vs. baseline 0.333（p=0.0156，统计显著，+0.467）。在约 495 题 BIRD benchmark 上，full pipeline 达到 EX=0.501 vs. baseline 0.474（+0.027），但差异未达统计显著性（p=0.136）；BIRD 结果应解读为方向性证据，而非确认的优势。P2 单轮模式将 BIRD token 成本从 32× 降至 7.9×，GIS 侧为 13.6×。

## 1. 引言

自然语言到 SQL 已成为大语言模型最活跃的应用方向之一，因为它为终端用户与结构化数据仓库之间提供了直接接口。近年来，在 Spider 和 BIRD 等公开 benchmark 上的性能取得了显著提升，表明现代语言模型在传统关系数据库场景下通常能够从自然语言问题合成可执行 SQL。然而，现有大多数 text-to-SQL 系统和评估数据集面向通用关系数据库设计，其核心挑战在于 schema 链接、连接推理、聚合和嵌套查询构造。相比之下，真实世界的地理空间数据库应用引入了这些 benchmark 中基本缺失的额外语义和操作复杂性。在地理空间场景中，查询频繁涉及几何感知的 schema grounding、空间谓词、坐标参考系、拓扑关系以及面积、距离、相交、缓冲等空间测量算子。这些特性使得地理空间 text-to-SQL 成为一个本质上不同的问题，而非通用仓库查询的简单扩展。

这种不匹配造成了实践与科学上的双重差距。从实践角度看，许多运营数据平台必须通过统一的自然语言接口同时服务地理空间与非地理空间工作负载。用户可能在同一系统中先问一个关于土地利用叠加的问题，再问一个关于建筑物邻近性的问题，然后又问一个关于表格业务统计的问题。从科学角度看，现有研究很少考察单一语义 grounding 架构能否在不产生严重领域偏差的情况下支持这些异构查询类型。大多数先前工作要么针对传统仓库式 SQL 生成进行优化，要么专注于特定领域的地理空间系统，而不评估超出 GIS 场景的可迁移性。因此，语义到 SQL 流水线的哪些组件能够跨域泛化、哪些组件编码了在某一场景有益但在另一场景有害的假设，目前仍不清楚。

这一挑战在 grounding 阶段尤为突出。在传统 text-to-SQL 系统中，schema 链接通常被定义为将用户表达与表名、列名或检索到的 schema 片段对齐的任务。在地理空间系统中，这一步骤更为复杂，因为模型还必须识别几何属性、推断相关空间算子、遵守空间参考变换，并区分拓扑、度量和基于聚合的推理。与此同时，激进的领域特定 grounding 策略在迁移到非地理空间数据库时可能引入意外偏差。一个针对中文 GIS 术语、空间算子提示和几何感知规则进行调优的系统，在空间 benchmark 上可能表现强劲，但如果其语义检索机制无法适应语言、schema 命名和算子分布的差异，则在英文仓库 benchmark 上可能表现不佳。理解这种张力对于构建鲁棒的跨域自然语言结构化数据接口至关重要。

为解决这一问题，本文提出 **NL2Semantic2SQL**，一种将语义层 grounding、schema 感知上下文构建、few-shot 检索、SQL 后处理和执行期自纠错集成到统一自然语言到 SQL 流水线中的跨域框架。框架的核心设计原则是双语跨域支持：该框架通过单一架构同时处理**中文 GIS 查询**和**英文仓库查询**。意图分类器双语运行，同时识别中文 GIS 关键词（如空间算子术语、中文行政区划名称）和英文仓库表达式（如"how many"、"what is the average"、"compare"）。这种双语路由不是事后补充，而是一个基本设计选择，使同一流水线能够在不进行语言特定模型切换的情况下服务于异构的真实世界部署。框架不将 SQL 生成视为单步提示问题，而是先解析语义上下文，再组装结构化 schema 证据，最后执行带有后验纠错的约束 SQL 生成。该设计的动机在于：GIS 查询和非 GIS 仓库查询共享对语义消歧的共同需求，但在必须向模型呈现的证据和约束类型上存在实质性差异。因此，本框架旨在保持统一架构的同时，允许显式而非隐式地纳入领域特定的 grounding 信号。

为评估问题的地理空间侧，本文进一步构建了一个基于执行的 GIS 导向 benchmark，涵盖空间算子覆盖和不同查询复杂度。该 benchmark 旨在捕捉主流 text-to-SQL 数据集中缺失的特征，包括空间选择、叠加分析、缓冲、空间聚合和几何敏感推理。然后，本文以基于 BIRD mini_dev 的仓库导向数据集对该 benchmark 进行跨域评估补充，在相同评估框架下直接比较 GIS 与非 GIS 场景。这种双轨设置使我们能够超越简单的"是否有效？"问题，转而提出更具信息量的科学问题：统一的语义到 SQL 架构在面对两个结构上不同的数据库领域时表现如何？

本文的贡献有三点。第一，提出了一种面向地理空间和传统仓库数据库跨域查询的统一 NL2Semantic2SQL 框架。第二，构建了一个专为空间 text-to-SQL 能力执行评估设计的 GIS benchmark。第三，通过跨域实验和误差分析，表明语义 grounding 在 GIS 场景中提供了显著优势，同时揭示了非 GIS 场景中的迁移局限性，特别是当检索和别名匹配策略编码了以 GIS 为中心的偏差时。这些发现表明，未来的 text-to-SQL 系统不应假设单一 grounding 策略能够在异构领域中均匀泛化，显式的跨域评估对于构建真正通用的自然语言数据库接口是必要的。

## 2. 方法

本文将跨域自然语言到 SQL 定义为将用户问题翻译为可执行 SQL 语句的任务，目标数据库可能属于地理空间或传统仓库领域。与标准 text-to-SQL 设置不同，本文问题定义中的目标数据库不假设共享同质的算子语义。在仓库式数据库中，正确的 SQL 生成主要依赖于 schema 链接、连接推理、聚合和时间过滤。然而在地理空间数据库中，模型还必须对几何列、空间关系、坐标系以及相交、包含、缓冲、面积或距离计算等领域特定算子进行推理。因此，本框架的目的不仅是生成有效的 SQL，还要提供一个语义中介层，使 grounding 过程适应每个领域的结构特征。

为支持这一设置，本文将 NL2Semantic2SQL 设计为三阶段流水线（图1：架构图）。在第一阶段，系统在语义层上执行语义解析，语义层存储表级和列级注释、别名、领域提示和任务相关元数据。在第二阶段，解析后的信息与数据库 schema 检查和可选的 few-shot 检索相结合，构建用于 SQL 生成的 grounded 上下文块。在第三阶段，生成的 SQL 经过后处理、在安全约束下执行，并在必要时通过执行反馈自纠错循环进行修订。这种分解将语义解释与 SQL 合成分离，使框架能够向语言模型暴露领域特定证据，同时保持最终生成步骤受可执行 schema 信息约束。

框架的核心设计特征是地理空间感知 grounding。对于地理空间表，系统显式识别几何列、几何类型和空间参考信息，并将算子级规则注入 grounding 上下文。这些规则区分拓扑谓词和度量谓词，并指定何时面积或距离计算需要坐标变换或地理投影转换。该机制旨在减轻语言模型从原始 schema 文本中推断低级地理空间执行约束的负担。同时，当空间信号缺失时，框架通过回退到通用 schema 链接和面向聚合的提示构建，保留了在非地理空间仓库数据库上运行的能力。从这个意义上说，框架在架构层面是统一的，但在语义 grounding 层面是自适应的。

对于非地理空间仓库查询，框架引入了三种领域自适应增强。第一，**跨域候选排序**根据解析的语义上下文是否包含空间算子或区域过滤器来重新排列候选来源的优先级。当此类信号缺失时，几何表被降低优先级，具有匹配列证据的表被提升。第二，**值感知 grounding** 将低基数文本列的样本不同值注入提示，使模型能够使用精确的分类值而非猜测字符串字面量。第三，**MetricFlow 式事实/维度建模**在语义模型注册表中存储每表角色（事实 vs. 维度）、实体列（主键或外键连接键）和度量列。在 grounding 时，候选表与该注册表匹配，通过对齐事实/维度对之间的共享实体推导出显式连接路径提示。这些提示仅针对非空间查询注入，保持 GIS 路径不变。

评估在双轨 benchmark 协议下进行。地理空间轨道使用一个 GIS 导向 benchmark，涵盖代表性空间查询模式，包括空间选择、叠加分析、度量距离查询、缓冲和空间聚合。非地理空间轨道基于 BIRD mini_dev 的仓库导向 benchmark，用于评估跨域泛化。对于两个轨道，均使用执行准确率（Execution Accuracy, EX）作为主要指标：当且仅当预测 SQL 的执行结果集与黄金 SQL 结果集在集合相等（含数值容差）下匹配时，预测被视为正确。

## 3. 实验结果

### 3.1 实验设置

**单轮模式（P2）说明**。在早期实验中，完整流水线通过 ADK agent 循环执行，由语言模型迭代调用工具（grounding、执行、重试）多个回合。这种多轮方法的 token 成本是基线的约 32 倍，并偶尔在某些题目上进入无界的 tool 调用循环。因此本文将 agent 循环替换为确定性的**单轮**流水线，内部命名为 **P2**：（1）本地构建 grounding 上下文（意图分类 + 语义解析 + prompt 拼装——无 LLM 调用）；（2）使用拼装好的 prompt 进行**单次** LLM 调用生成 SQL；（3）后处理 SQL（安全检查、标识符引用、意图门控的 LIMIT 注入）并执行；若执行失败，使用快速纠错 LLM 调用重试（最多 T=2 次）。P2 模式将 BIRD token 成本从约 32× 降至 7.9×，同时将执行准确率从 0.450 提升至 0.501，因为它消除了 agent 循环的随机性，确保每道题恰好通过同一套 grounding 流水线一次。本文中 BIRD 的所有结果均使用 P2 模式；GIS 的完整流水线结果使用 ADK agent 循环（受益于多工具空间分析能力），但同样的 P2 架构作为可选方案存在。

本文在双轨 benchmark 协议下评估 NL2Semantic2SQL。GIS 轨道使用 100 题 benchmark（从 20 题试点扩展而来），涵盖四个难度级别：Easy（24 题）、Medium（36 题）、Hard（25 题）和 Robustness（15 题）。仓库轨道使用 BIRD mini_dev V2 中约 495 题（单轮 P2 模式），跨越导入 PostgreSQL 的 11 个数据库 schema 的三个难度级别（simple、moderate、challenging）。两个轨道均使用执行准确率（EX）作为主要指标：当且仅当预测 SQL 的执行结果集与黄金 SQL 结果集在集合相等（含数值容差）下匹配时，预测被视为正确。本文报告所有 EX 值的 95% Wilson 置信区间和 McNemar 配对显著性检验。

对于每个轨道，比较两种模式：
- **基线（Baseline）**：直接 LLM 生成（Gemini 2.5 Flash），仅使用 schema dump，无语义 grounding。
- **完整流水线（Full pipeline）**：NL2Semantic2SQL，含语义层解析、schema 感知上下文构建、few-shot 检索（仅 GIS 轨道）、SQL 后处理和执行期自纠错（BIRD 使用单轮模式）。

此外，本文还将 **DIN-SQL** [3] 作为两个轨道上的外部基线进行比较。

### 3.2 GIS 轨道结果

表 1a 报告了扩展后 100 题 benchmark 的 GIS 空间执行准确率（Spatial EX，85 道普通空间 SQL 题），表 1b 报告了鲁棒性成功率（Robustness Success Rate，15 道安全/拒绝题）。本文分开报告这两类指标，因为它们衡量的是本质不同的能力：Spatial EX 衡量 SQL 生成准确性，而 Robustness Success Rate 衡量安全执行和拒绝行为。将两者合并为单一"Overall EX"指标是非标准的，会混淆两个不同的评估维度；本文仅将综合数字作为次要参考保留。

**Benchmark 构建**：100 题 GIS benchmark 的构建过程如下：（1）由领域专家手工编写 20 道种子题，涵盖 4 个难度级别的 16 个空间查询类别；（2）使用 Gemini 2.5 Flash 以实际 CQ 数据库 schema 为上下文生成 80 道额外题目，然后通过在实时 PostgreSQL/PostGIS 数据库上执行每道黄金 SQL 进行验证；（3）6 道鲁棒性题目的黄金 SQL 为空（测试拒绝行为）；（4）所有 94 道可执行黄金 SQL 均经验证返回非空、正确结果。Benchmark 和黄金 SQL 以 `benchmarks/chongqing_geo_nl2sql_100_benchmark.json` 发布。

**表 1a：GIS 空间执行准确率——85 道普通空间 SQL 题**

| 难度 | N | 基线 EX [95% CI] | 完整流水线 EX [95% CI] | Delta |
|------|---|-----------------|----------------------|-------|
| **Spatial EX（全部）** | **85** | **0.529 [0.424, 0.632]** | **0.682 [0.577, 0.772]** | **+0.153** |
| Easy | 24 | 0.750 [0.551, 0.880] | 0.833 [0.641, 0.933] | +0.083 |
| Medium | 36 | 0.389 [0.248, 0.551] | **0.694 [0.531, 0.820]** | +0.305 |
| Hard | 25 | 0.520 [0.335, 0.700] | 0.520 [0.335, 0.700] | 0.000 |
| McNemar | 85 | b=4, c=17 | **p=0.0072** | 显著（α=0.05） |
| 平均 token 数 | 85 | 753 | 10,261（13.6×） | |

**表 1b：GIS 鲁棒性成功率——15 道安全/拒绝题**

| 指标 | N | 基线 [95% CI] | 完整流水线 [95% CI] | Delta |
|------|---|--------------|-------------------|-------|
| **Robustness Success Rate** | **15** | **0.333 [0.152, 0.583]** | **0.800 [0.548, 0.930]** | **+0.467** |
| McNemar | 15 | b=0, c=7 | **p=0.0156** | 显著（α=0.05） |

*次要参考*：合并全部 100 题得到 Overall EX=0.500（基线）vs. 0.700（完整流水线），Δ=+0.200，McNemar p=0.0002。该综合数字仅供参考；主要指标为上述分开报告的 Spatial EX 和 Robustness Success Rate。

完整流水线在 Spatial EX 上比基线高出 +0.153（McNemar p=0.0072，在 α=0.05 下统计显著）。最大空间增益出现在 Medium 难度题目（+0.305）。Medium 题目从语义 grounding 中获益最多，因为它们涉及多步空间推理，几何感知类型注入和意图条件化算子路由能够解决基线无法处理的执行期类型错误。

Hard 题目无变化（两者均为 0.520）：完整流水线在部分 Hard 题目上的增益被其他题目的回退所抵消，主要是意图误分类将错误的 spatial_join 上下文引入邻近缓冲查询的情况。

在 Robustness 轨道上，完整流水线达到 0.800 vs. 基线 0.333（+0.467，McNemar p=0.0156，统计显著）。鲁棒性题目受益于安全后处理，该处理捕获了基线 LLM 否则会执行的危险操作（DELETE、UPDATE、DROP）。例如，问题"删除所有未命名的道路"测试系统是否正确拒绝破坏性 DELETE 操作。基线生成 `DELETE FROM cq_osm_roads_2021 WHERE name IS NULL`，这会修改数据。完整流水线的安全后处理器拦截写入意图并返回规范拒绝消息。

GIS 轨道的 token 成本相对于基线为 13.6×（每题平均 10,261 vs. 753 个 token）。这一成本因 GIS 领域的安全和精度要求而合理：几何感知 grounding 防止错误的单位计算，鲁棒性执行防止危险的 schema 修改。

作为参考，早期 500 题多轮运行显示：基线 0.458，完整流水线 0.450（-0.008，McNemar p=0.8151）。P2 单轮模式不仅降低了成本，还消除了小幅多轮退化：完整流水线现在在 BIRD 轨道上也超过了基线。

### 3.4 DIN-SQL 外部基线对比

本文将 DIN-SQL [3] 作为两个轨道上的外部提示基线进行比较。DIN-SQL 使用带自纠错的分解式上下文学习。GIS 上的 DIN-SQL 评估现在涵盖与基线和完整流水线相同的 100 题 benchmark，实现了公平的直接比较。

**表 3：DIN-SQL 外部基线对比（GIS 100 题，BIRD 约 495 题）**

| 轨道 | N | 基线 EX | DIN-SQL EX | 完整流水线 EX | 完整 vs. DIN-SQL |
|------|---|---------|------------|-------------|----------------|
| GIS Spatial 85q | 85 | 0.529 | 0.565 | **0.682** | +0.118 |
| GIS Robust. 15q | 15 | 0.333 | 0.267 | **0.800** | +0.533 |
| BIRD ~495q | ~495 | 0.474 | 0.482 | **0.501** | +0.019 |

**在 GIS Spatial 轨道（85 题）上**：DIN-SQL 达到 EX=0.565，仅略高于直接 LLM 基线（0.529，McNemar DIN-SQL vs. 基线 p=0.508，不显著）。这证实了不带空间 grounding 的分解提示对地理空间 SQL 不能提供比直接生成更有意义的改进。完整 NL2Semantic2SQL 流水线达到 Spatial EX=0.682，比 DIN-SQL 统计显著提升 **+0.118**（配对 McNemar b=3，c=13，**p=0.0213**，在 α=0.05 下显著）。

**在 GIS Robustness 轨道（15 题）上**：DIN-SQL 仅达到 0.267 成功率，略低于基线（0.333），因为 DIN-SQL 没有内置的安全/拒绝处理机制。完整流水线达到 0.800，比 DIN-SQL 统计显著提升 **+0.533**（配对 McNemar b=0，c=8，**p=0.0078**，显著）。

**在 BIRD 仓库轨道上**：DIN-SQL 达到 EX=0.482（有效性=0.990）。完整流水线 EX=0.501 在数值上高于 DIN-SQL EX=0.482（+0.019），但差异**不具统计显著性**（McNemar p=0.382）。三个系统——基线（0.474）、DIN-SQL（0.482）和完整流水线（0.501）——在 BIRD 轨道上统计上相当。

总结而言，完整流水线在 GIS 轨道（Spatial 和 Robustness 均）上相对于基线和 DIN-SQL 均获得统计显著的增益，而在 BIRD 上三个系统统计上相当。这一模式强化了本文的核心论点：带意图条件化路由的语义 grounding 在几何感知 schema 链接、SRID/坐标处理和安全执行至关重要的地理空间 SQL 中产生领域特定增益，而 DIN-SQL 等纯提示工程基线在传统仓库 SQL 上仍具竞争力。

### 3.5 Token 成本分析

完整流水线由于语义 grounding 上下文注入而产生比基线高得多的 token 成本。表 4 报告了每题平均 token 数。

**表 4：每题平均 token 数**

| 轨道 | N | 基线 token | 完整流水线 token | 比率 | DIN-SQL token |
|------|---|-----------|----------------|------|--------------|
| GIS | 100 | 753 | 10,261 | 13.6× | 未追踪 |
| BIRD | ~495 | 1,010 | 7,975 | 7.9× | 未追踪 |

GIS token 比率（13.6×）高于 BIRD（7.9×），因为 GIS grounding 上下文包含几何类型注释、SRID 信息、空间算子规则和 few-shot 示例，这些在仓库轨道上均不存在。

对于 BIRD 轨道，P2 单轮模式将 token 比率从约 32×（早期实验中的多轮 agent runner）降至 7.9%。这一降低通过将每道题恰好路由一次通过 grounding 流水线而非使用迭代 agent 循环实现，同时通过消除 agent runner 挂起也提升了 EX。

**部署考量**：7.9–13.6× 的 token 开销在规模化时不可忽视。对于安全关键的 GIS 场景（几何精度、数据修改防护），开销显然合理。对于仓库查询，7.9× 开销是否合理取决于应用场景：在延迟敏感或成本受限的部署中，方向性的 +0.027 EX 增益可能不值得额外成本。一种有前景的缓解措施是选择性 grounding：不为每个查询注入完整语义上下文，而是先执行轻量级意图分类（已实现），然后仅检索最相关的 top-K schema 元素和 grounding 规则。初步分析表明，对于聚合意图查询，grounding 提示可以减少约 60% 而不影响生成的 SQL，因为大多数聚合查询不需要空间算子提示。

### 3.7 误差分析

本文将完整流水线在 BIRD 轨道上的失败分为三类。执行有效性=0.996（约 495 题），仅约 2 题（约 0.4%）产生无效 SQL；绝大多数失败是语义错误但执行成功却返回错误结果的 SQL。

| 错误类型 | 失败占比 |
|---------|---------|
| 结果错误（有效 SQL，答案错误） | ~95% |
| 未生成 SQL（gen_status=no_sql） | ~5% |
| 无效 SQL（执行错误） | <1% |

主要失败模式是语义错误但执行成功却返回错误结果的 SQL。人工检查揭示了三种反复出现的模式：

1. **连接路径混淆**（约 40%）：模型在事实表和维度表之间选择了错误的连接路径。
2. **聚合语义**（约 30%）：模型在黄金 SQL 使用 COUNT(*) 的地方应用了 COUNT(DISTINCT ...)，或反之。
3. **日期/时间解析**（约 25%）：BIRD 数据集使用非标准日期格式，模型解析不一致。

P2 单轮模式通过几乎消除无效 SQL 显著提升了有效性（0.978→0.996）；单轮模式将每道题恰好路由一次通过 grounding 流水线，避免了多轮模式中产生空或格式错误 SQL 的 agent 循环挂起。

## 4. 讨论

### 4.1 语义 Grounding 为何在 GIS 中有效——以及为何优势现在具有统计显著性

扩展后的 100 题 GIS benchmark 证实了 20 题试点所暗示但无法统计证实的结论：带意图条件化路由的语义 grounding 为地理空间 SQL 提供了大而可靠的优势。两个主要 GIS 指标均具统计显著性：Spatial EX +0.153（McNemar p=0.0072）和 Robustness Success Rate +0.467（p=0.0156）。三种机制驱动了这一改进：

第一，**几何感知类型注入**确保模型知道哪些列是几何列、其 SRID 是什么，以及何时需要地理投影转换。没有这一机制，基线频繁遗漏 `::geography` 转换，产生以度为单位而非米为单位的面积/距离值。该机制驱动了大部分 Medium 难度改进（+0.305）。

第二，**意图条件化算子路由**将领域特定规则（KNN `<->` 算子、LIMIT 注入）限定于相关查询。这消除了在 Phase A 之前流水线中导致退化的误注入。

第三，**安全和鲁棒性执行**通过 SQL 后处理捕获危险操作（DELETE、UPDATE）并处理拒绝/反幻觉情况。基线 LLM 没有此类防护，在鲁棒性套件上仅得 0.333，而完整流水线达到 0.800（+0.467）。

从 p=0.125（n=20）到 p=0.0072（Spatial，n=85）和 p=0.0156（Robustness，n=15）的转变说明了一个方法论要点：效果是真实的，但在试点中统计功效不足。100 题 benchmark 提供了足够的功效来确认两个优势都不是抽样伪影。

组件归因分析（第 3.8 节表格）证实语义 grounding 是 GIS 改进的主要贡献者，占总 +0.200 EX 增益的 65%。这解决了改进可能仅归因于安全防护等偶然因素的担忧。虽然安全执行贡献了有意义的 +0.070（35%），但大部分增益来自几何感知 schema grounding，帮助模型正确应用 `::geography` 转换、空间谓词和坐标系处理——这些是基线和 DIN-SQL 均不提供的能力。

### 4.2 BIRD 结果：方向性但不显著

在 BIRD 轨道上，完整流水线达到 EX=0.501 vs. 基线 0.474（+0.027，McNemar p=0.136）。置信区间重叠，结果在 α=0.05 下不具统计显著性。本文明确声明：BIRD 结果应解读为方向性证据，而非确认的优势。

完整流水线 EX=0.501 在数值上高于 DIN-SQL EX=0.482（+0.019），但差异不具统计显著性（McNemar p=0.382）。三个系统——基线（0.474）、DIN-SQL（0.482）和完整流水线（0.501）——在 BIRD 轨道上统计上相当。完整流水线相对于基线和 DIN-SQL 的优势均未在当前样本量下达到统计显著性。

执行有效性的改进（0.978 → 0.996）值得注意：完整流水线几乎消除了仓库查询上的无效 SQL，即使总体 EX 没有显著改善，这也是一个实际好处。

### 4.3 Token 成本是真实的部署考量

13.6× GIS 和 7.9× BIRD 的 token 开销不可忽视。本文诚实地讨论这一点，而非轻描淡写。P2 单轮模式将 BIRD token 成本从约 32× 降至 7.9×，同时提升了 EX——证明架构优化（消除多轮 agent 循环）可以在不牺牲质量的情况下降低成本。

对于 GIS 轨道，13.6× 开销因领域性质而合理：安全关键的空间分析需要几何元数据、算子规则和安全执行，这些自然会增大上下文。对于 BIRD 轨道，7.9× 开销是否合理取决于应用场景：在延迟敏感或成本受限的设置中，方向性的 +0.027 EX 增益可能不值得额外成本。

实际部署策略：对所有 GIS 查询使用完整流水线（安全需要），仅在质量关键（如财务报告）而非探索性场景下对 BIRD 查询使用完整流水线。

### 4.4 MetricFlow 式建模的效果

误差分析表明，仓库性能的主要瓶颈不是 SQL 语法生成，而是语义 schema 导航。MetricFlow 式建模通过声明实体（连接键）、度量（可聚合事实）和维度（描述性属性）直接针对这一问题。本文通过为最易出错的 BIRD schema 注册 MetricFlow 式元数据并将推导的连接路径提示注入 grounding 提示来验证这一假设。这一单一干预在 50 题试点上将总体执行准确率从 0.520 提升至 0.540，moderate 题目从 0.316 恢复至 0.368，challenging 题目从 0.333 提升至 0.500。本文将此解读为仓库侧性能受缺失结构元数据而非表面形式生成瓶颈的直接证据，有针对性的语义建模在不改变底层语言模型的情况下弥合了大部分差距。

### 4.5 跨语言观察（初步压力测试）

本文将跨语言实验定位为初步压力测试，而非完全受控的跨语言评估。翻译题目由 LLM 翻译器（Gemini 2.0 Flash）生成，未经母语人工标注者验证；此外，表名和列名保留原始英文形式，这降低了与真正双语 schema 相比的跨语言 schema 链接难度。在这一受限设置下，框架的多语言别名机制足以在题目被改写为中文时恢复大多数 schema 引用，执行准确率（0.500 vs. 0.540）和有效性（0.900 vs. 0.940）仅损失四个百分点。剩余差距集中在 moderate 和 challenging 题目上，表明聚合措辞和连接键引用中的改写引起的歧义是主要残余错误。本文不声明一般性的跨语言鲁棒性，更严格的评估需要与（i）无别名基线、（ii）翻译回英文基线和（iii）人工翻译题目进行比较。

### 4.5a 与 GeoSQL-Eval 的关系

本文工作与 GeoSQL-Eval [13] 互补。GeoSQL-Eval 提供跨 340 个 PostGIS 函数的细粒度函数级评估（语法正确性、参数匹配、返回类型识别），而本框架评估带语义 grounding 和跨域泛化的端到端查询生成。未来工作可将 GeoSQL-Bench 的函数级指标集成到本评估协议中，提供宏观级（EX）和微观级（函数命中率、参数准确性）评估。

### 4.6 局限性

需要注意几个局限性。第一，100 题 GIS benchmark 仅基于重庆空间数据；对其他空间数据库约定（GeoPackage、Shapefile schema、非 WGS84 坐标系）和非中文 GIS schema 的泛化尚未评估。第二，在 BIRD 轨道上，完整流水线相对于基线的 +0.027 EX 优势方向性明确但不具统计显著性（p=0.136）；与 DIN-SQL 的三方比较同样未显示统计显著差异。第三，跨语言实验使用 LLM 翻译题目，未经人工验证；完整的跨语言声明需要人工标注翻译。第四，token 成本（基线的 8–14×）是真实的部署考量，特别是对于边际准确率增益统计上不确定的仓库查询。第五，当前评估使用 Gemini 2.5 Flash；在其他 LLM 系列（如 GPT-4o、Claude、开源模型）上的行为可能不同。第六，本文基线仅为直接 LLM+schema dump 和 DIN-SQL；与 MAC-SQL 和其他近期方法的比较对于将本方法定位于完整最新技术水平是必要的。第七，基于执行的评估将任何结果集不匹配视为失败，即使预测 SQL 在语义上等价但在排序或数值精度上不同。第八，当前 grounding 规则特定于 PostgreSQL/PostGIS 语法（如 `::geography` 转换、`<->` KNN 算子、`ST_*` 函数族）。适配其他空间数据库（MySQL Spatial、Oracle Spatial、SpatiaLite）需要替换意图路由器和后处理器中的方言特定规则字典，但总体架构（语义层 → 意图分类 → grounding → 生成 → 后处理）是方言无关的。

### 4.7 未来工作

本研究提出三个方向：
1. **跨域 GIS benchmark 扩展**。将 GIS benchmark 扩展到涵盖多个地理空间数据库（非中文、不同坐标系、栅格数据），并增加每类别题目数量以进行更细粒度的消融分析。
2. **BIRD 统计显著性**。将 BIRD 评估扩展到 1000+ 题，以获得足够的功效来确认或否定观察到的 +0.027 方向性优势。
3. **Token 成本降低**。研究选择性 grounding（仅注入相关语义上下文块而非完整元数据）和提示压缩，以在保持 EX 增益的同时降低 7.9–13.6× 的 token 开销。

## 5. 相关工作

### 5.1 基于大语言模型的 Text-to-SQL

大语言模型在 text-to-SQL 上的应用通过日益复杂的提示工程策略迅速进展。早期的上下文学习方法证明，当提供 schema 描述和示例查询对时，LLM 能够从自然语言问题生成可执行 SQL [1, 2]。DIN-SQL [3] 引入了分解提示流水线，将 schema 链接、查询分类和 SQL 生成分为不同阶段，并以自纠错作为后生成步骤。DAIL-SQL [4] 基于 SQL 骨架相似性系统化了 few-shot 示例选择，以最小 token 成本在 Spider benchmark 上取得了强劲结果。C3 [5] 探索了带校准偏差纠正的零样本 ChatGPT text-to-SQL，展示了不依赖手工 few-shot 示例的互补提示工程解决方案。近年来，多 agent 和工具增强架构也已出现，其中专门的 agent 将 schema 检索、SQL 生成和执行验证作为独立的协作角色处理。本框架与 DIN-SQL 共享分解理念，但增加了显式语义层作为用户问题和 schema 之间的结构化中介，而非仅依赖提示级 schema 链接。

### 5.2 Text-to-SQL Benchmark

Spider [1] 建立了标准的跨数据库 text-to-SQL benchmark，采用基于执行的评估，引入了跨未见数据库 schema 泛化的挑战。BIRD [6] 将这一设置扩展到更大、更真实的数据库，采用值感知评估和难度分层，揭示 LLM 性能在复杂聚合和多跳连接查询上显著下降。Spider 2.0 [7] 进一步扩展范围，包括 dbt 转换和方言变体等企业式工作流。BEAVER [8] 专门针对具有私有 schema 和复杂业务逻辑的企业数据仓库，表明最先进模型在真正私有企业数据上的准确率接近零。本文工作的动机是一个互补的差距：主流 text-to-SQL benchmark（Spider、BIRD、Spider 2.0）对可执行 PostGIS 式空间 SQL、几何感知 schema grounding 和 SRID/坐标系处理的覆盖有限，且没有一个在单一评估协议中评估 GIS 和仓库查询语义之间的跨域迁移。

### 5.3 地理空间自然语言接口

地理空间数据库的自然语言接口与通用 text-to-SQL 有着不同的研究脉络。早期地理空间问答工作，如 GeoQuery [Zelle 1996]，使用归纳逻辑编程学习地理数据库查询解析器，专注于小型地理事实数据库。Punjani 等人 [9] 将其扩展到语义网上链接地理空间数据的基于模板的问答，证明当适当的本体映射可用时，结构化空间知识可以通过自然语言查询；他们的工作强调了空间谓词覆盖和坐标系感知的重要性，这对 PostGIS 式 SQL 生成仍然相关。近期工作探索了使用 LLM 进行空间推理和地理知识提取 [10]，但对 PostGIS 或空间扩展关系数据库上 text-to-SQL 能力的系统评估仍然稀缺。GeoSQL-Eval [13] 通过涵盖 340 个 PostGIS 函数的 14,178 个任务的综合 PostGIS 函数级评估框架解决了这一差距。本文工作在关注点上有所不同：GeoSQL-Eval 评估单个 PostGIS 函数调用和语法正确性，而 NL2Semantic2SQL 针对带语义层 grounding、意图路由和跨 GIS 与仓库领域安全执行的端到端跨域查询生成。本文 GIS benchmark 旨在以端到端执行准确率补充这一函数级评估。

### 5.4 语义层与 Schema Grounding

语义层作为分析查询和原始数据库 schema 之间中介的概念起源于商业智能工具，并在 MetricFlow [11] 和 dbt 语义层等框架中得到形式化。这些系统将实体、维度、度量和指标定义为声明式元数据，控制连接和聚合的构建方式。在 text-to-SQL 上下文中，语义层提供了一种原则性方式，为 LLM 提供关于 schema 语义的结构化证据，而非要求模型仅从原始 DDL 推断连接路径和聚合规则。本框架将这一概念适配于跨域使用，将 GIS 特定注释（几何类型、SRID、空间算子）与仓库式实体关系元数据相结合，通过统一 grounding 架构支持两个查询领域。

### 5.5 跨域与跨语言 Text-to-SQL

跨域泛化主要通过在相同语言和算子领域内跨数据库 schema 的零样本迁移进行研究 [1, 6]。跨语言 text-to-SQL（问题以一种语言提出，schema 以另一种语言定义）受到的系统关注较少，尽管 CSpider [12]（中文 Spider）等多语言 benchmark 证明语言不匹配引入了额外的 schema 链接挑战。本文工作将跨域维度从 schema 变化扩展到包括算子异质性（空间 vs. 表格）和语言异质性（中文问题对应英文仓库 schema），提供了对泛化格局更全面的视角。

## 6. 结论

本文提出了 NL2Semantic2SQL，一种面向地理空间和传统仓库数据库自然语言接口的跨域框架。该框架将语义层 grounding、意图条件化路由、schema 感知上下文构建、few-shot 检索、SQL 后处理和执行期自纠错集成到单一三阶段流水线中。框架的核心设计原则是双语跨域支持：通过双语运行的意图条件化路由处理中文 GIS 查询和英文仓库查询，无需语言特定的模型切换。本文构建了一个带独立鲁棒性套件的 100 题 GIS benchmark，并将其与约 495 题 BIRD mini_dev 子集相结合，定义了双轨评估协议。

四个发现突出。第一，在 85 题 GIS Spatial 轨道上，完整流水线达到 Spatial EX=0.682 vs. 基线 0.529（McNemar p=0.0072，在 α=0.05 下统计显著，+0.153）。改进由几何感知 grounding（Medium 上 +0.305）和意图条件化算子路由驱动。第二，在 15 题 GIS Robustness 轨道上，完整流水线达到成功率=0.800 vs. 基线 0.333（McNemar p=0.0156，统计显著，+0.467），由安全后处理驱动。两个 GIS 结果均显著强于 20 题试点（p=0.125），证实效果是真实的但在早期评估中统计功效不足。第三，在约 495 题 BIRD benchmark（单轮 P2 模式）上，完整流水线达到 EX=0.501 vs. 基线 0.474（+0.027）。BIRD 优势方向性明确但不具统计显著性（McNemar p=0.136）；三个系统——基线（0.474）、DIN-SQL（0.482）和完整流水线（0.501）——在仓库查询上统计上相当。BIRD 结果应解读为方向性证据，而非确认的优势。第四，P2 单轮模式将 BIRD token 成本从约 32× 降至 7.9×，同时提升了 EX，证明架构优化可以同时降低成本和提升质量。

综合来看，结果表明带双语意图条件化路由的统一语义到 SQL 架构为 GIS 空间准确性（p=0.0072）和鲁棒性（p=0.0156）提供了统计显著的增益，为仓库查询提供了方向性增益。Token 成本（基线的 8–14×）是真实的部署考量，特别是对于边际准确率增益统计上不确定的仓库查询；未来工作应通过选择性 grounding 或上下文压缩来解决这一问题。
